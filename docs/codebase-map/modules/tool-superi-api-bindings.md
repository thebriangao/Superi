## Purpose and ownership

`superi-api-bindings` owns deterministic publication and drift checks for the committed
desktop-facing TypeScript representation of the canonical `superi-api` surface. It is a repository
utility, not a runtime crate, transport, application shell, or state owner.
The current artifact includes the strict extension registry query, full replacement event and
resource, exact identities, lifecycle, capabilities, safe failure, features, stable public control
reference, all twelve track mutation tags, all six marker mutation tags with complete create state,
all seven multicam mutation tags, track, marker, and multicam action evidence, typed audio-video
link, synchronization, detach, and complete routing, typed track output
graph state, project editor schema `1.7.0`,
the additive exact `set_transition` and `retime` timeline operations used by the desktop timeline
inspectors, and strict existing-child placement and selection-derived
compound request and per-track identity DTOs, their project action variants, and typed placement and
compound result evidence for the desktop nesting consumer.

## Source inventory

- `open/tools/superi-api-bindings/Cargo.toml`: Declares one workspace tool and enables the API's
  generator-only `typescript-bindings` feature.
- `open/tools/superi-api-bindings/src/lib.rs`: Resolves the canonical artifact independently of the
  caller's working directory, delegates pure rendering to `superi-api`, compares exact bytes without
  mutation, and writes fresh deterministic output only when needed.
- `open/tools/superi-api-bindings/src/main.rs`: Exposes exact `generate` and nonmutating `check`
  commands with explicit current, missing, stale, written, and unchanged outcomes.
- `open/tools/superi-api-bindings/tests/generator_contract.rs`: Proves deterministic idempotent
  generation, byte-preserving stale and missing checks, and exact agreement between the committed
  artifact and fresh API-owned output.

## Public surface

The command surface is:

```text
superi-api-bindings generate
superi-api-bindings check
```

The library exports `CANONICAL_BINDINGS_PATH`, `canonical_bindings_path`, `render`, `check_path`,
`generate_path`, `CheckStatus`, `GenerateStatus`, and `BindingToolError`. `check_path` is read-only
and distinguishes current, missing, and stale state. `generate_path` creates parent directories as
needed, preserves a current file without rewriting it, and replaces only the explicitly supplied
generated artifact with fresh output.

## Architecture and data flow

The tool owns filesystem policy while the API owns schema meaning:

```text
superi-api canonical registries and public DTOs
  -> pure render_typescript_bindings
  -> exact generated bytes
  -> check_path for read-only drift evidence
  -> or generate_path for the canonical committed artifact
```

The canonical location is `open/bindings/typescript/superi-api.ts`. It is resolved from
`CARGO_MANIFEST_DIR`, so generation and checking do not depend on the process working directory.
No timestamp, checkout path, environment identity, network response, or existing artifact content
enters generated bytes.
Extension declarations therefore come from the same Rust DTO and registry owner as every other
public type. The tool does not infer a plugin, start a worker, grant a capability, or create a
privileged engine route.

## Dependencies and consumers

- `superi-api` supplies the pure deterministic renderer behind its opt-in TypeScript feature.
- The Rust standard library supplies argument handling, paths, filesystem access, errors, and exit
  status. The tool has no network, Tauri, UI, engine, project, or lower-subsystem dependency.
- Cargo workspace tests consume the current-artifact contract automatically.
- Contributors use `generate` after public DTO or registry changes and `check` during verification.
- `ci/frontend-smoke` consumes the committed result through strict typechecking and a production
  Vite bundle, but it does not invoke or depend on this Rust tool at runtime.

## Invariants and operational boundaries

- `check` never creates a directory or file and never rewrites current, stale, or missing state.
- `generate` writes only the caller-supplied target, and the command supplies only the canonical
  artifact path.
- Current bytes are not rewritten, which preserves file metadata and makes repeated generation
  idempotent.
- The tool never reconstructs methods, events, resources, error types, or TypeScript declarations.
  Registry parity and serialization meaning remain owned and validated by `superi-api`.
- Generated output is ordinary committed source for desktop clients, not authoritative project
  state, a hidden intelligent-feature channel, a transport implementation, or proof of a UI.
- No command downloads dependencies, opens a socket, starts an engine, mutates a project, or touches
  the closed tree.

## Tests and verification

Three integration contracts create exact process-local temporary directories. They prove first
generation writes fresh bytes, a second generation is unchanged, output equals a separate pure
render, stale and missing checks are classified correctly, stale bytes remain byte-identical after
checking, and the committed canonical artifact equals a fresh render. Fresh focused package tests
and the direct locked `check` command passed in this checkpoint.

The API-owned contract independently proves complete canonical method, event, and resource coverage,
required project, command-log, AI, extension, and track declarations, deterministic output, and
absence of timestamps and checkout paths. The generated artifact includes the command-log query and
result union, record types, script step, track action union, semantic evidence, graph output state,
event correlation sequence, method map, resource map, and timeline edit union. That union includes
`set_transition` with stable timeline, track, transition, and exact duration fields, while the
project action union includes `place_nested_sequence` and `create_compound_clip` with every nested
DTO and evidence variant. It also includes `mutate_multicam`, complete source, sync, attach, switch,
cut, audio, and detach DTOs, and `multicam_mutated` evidence. Frontend proof belongs to the workspace
consumer and is not inferred from this tool's Rust tests.

## Current status and risks

The generator and drift check are implemented for the complete current catalog. The tool publishes
one file and intentionally provides no watch mode, automatic application binding, transport adapter,
or package publication. Specta is a generator-only dependency selected through the API feature; the
default runtime API and engine graph remain unchanged.
Extension, track, multicam, nesting, and compound declarations remain compile-time contracts only. The
production application supplies the separate generated-client and durable native-route consumer
proof.

Direct filesystem writes can still fail because of permissions, storage exhaustion, or concurrent
external modification. Such failures are explicit and never reported as current. The generated
artifact is recoverable by rerunning `generate` from canonical Rust source.
