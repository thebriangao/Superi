# P2.W04.C001

## Outcome

Implemented an independent audio processing graph with deterministic editable topology and a
separately prepared real-time processing plan. The public graph preserves exact sample coordinates,
ordered channel meaning, explicit routes, bounded block sizes, and audible continuity across
successive blocks.

## Architecture and integration

`superi-audio::graph` owns audio-specific graph, node, and edge identities instead of reusing the
image and GPU graph's domains. `AudioGraph` stores nodes and edges in ordered collections, rejects
missing endpoints, duplicate identities, ambiguous node inputs, unequal ordered channel layouts,
and direct or transitive cycles before mutation, and computes a stable topological order.

Each graph fixes one integral sample rate and positive maximum frame count. Nodes declare one
optional input layout and one output layout. A node without input is a source; a processing node
accepts exactly one route in this foundational graph. No edge silently resamples, upmixes,
downmixes, remaps, pans, or sums channels.

`AudioGraph::prepare` resolves one destination and its connected ancestors, rejects unconnected
required inputs and missing processor implementations, computes one immutable processing order,
and fallibly preallocates every interleaved f32 intermediate buffer. Preparation is separate from
editing, so the audio callback cannot mutate topology or allocate graph storage.

`PreparedAudioGraph::process` requires `ExecutionDomain::Audio`. It validates sample rate, positive
bounded frame count, exact output size, coordinate overflow, and adjacency with the prior
successful block before invoking processors. The graph-owned successful path takes no lock,
allocates no memory, and frees no memory; failure-only diagnostics preserve shared classified
errors with graph and node context.

The graph composes `superi-core`'s `SampleTime` and ordered `ChannelLayout` and
`superi-concurrency`'s platform-owned audio-domain policy. It does not add a media-I/O dependency,
device backend, playback loop, bus mixer, resampler, meter, A/V coordinator, or plugin host.

## Implementation process

The public integration contract was added first and failed because the audio graph module exposed
only scaffold documentation. The implementation then added typed topology and validation,
destination-scoped preparation, processor blocks, fixed intermediate storage, and exact block
execution. The first green review found unconditional diagnostic-context construction on the
successful callback path; that allocation was moved behind failure branches before broader proof.

Primary-source review used the W3C Web Audio block-processing and explicit-channel model plus
Apple Audio Unit guidance that render resources and stream formats are prepared before cached
real-time render calls. Repository law remained stricter: all graph-owned buffers are prepared
outside the callback, and the callback is explicitly bound to Superi's allocation-free audio
domain.

## Verification

The public `audio_graph_contract` builds a source-to-gain-to-gain chain and processes two adjacent
48 kHz stereo blocks through `PreparedAudioGraph` on `ExecutionDomain::Audio`. It verifies every
channel-distinct interleaved f32 result, the stable node order and destination layout, and the exact
next sample after both blocks.

The same contract proves atomic cycle, duplicate-input, channel-layout, and missing-endpoint
rejection; unconnected-input preparation failure; audio-domain enforcement; and nonadvancing
failure for mismatched rates, oversized blocks, incorrect output length, and sample discontinuity.
Package tests, warnings-denied Clippy, generated public documentation, dependent engine tests,
formatting, map validation, and the final repository verifier are recorded in the ignored execution
log with their final results.

## Remaining constraints

The foundational graph accepts one input per processing node. Multi-input summing, buses, channel
maps, gain, fades, pan, mute, solo, automation, and advanced routing remain later mixing work.
Caller-provided processors must honor the documented nonblocking and allocation-free success-path
contract; this graph cannot inspect their internals. There is no decoder, engine playback, device,
physical latency, A/V synchronization, resampling-quality, metering, long-session, or plugin-host
proof in this checkpoint.
