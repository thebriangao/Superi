# P2.W06.C003

## Outcome

Implemented foreground playback orchestration across decoded-frame provenance, immutable graph
evaluation, complete frame-cache identity, display color execution, bounded audio-device admission,
the shared playback clock, and a lossless viewport handoff. One exact frame is evaluated at a time
on a playback-priority worker, paced by the real audio presentation clock, and retained for retry
when the viewport is saturated.

Decoded representation, timestamp, duration, typed metadata, source color history, and source alpha
meaning remain immutable provenance beside the processed scene value. The cacheable scene preserves
its own exact nonterminal color history and alpha association, while the viewport branch applies an
explicit display transform and publishes separate terminal color metadata without changing the
cached scene result.

## Architecture and integration

`PlaybackSceneFrame<V>` is the graph and cache value. It carries exact output timing, decoded input
provenance, scene color history, alpha meaning, and a generic scene payload. Its decoded snapshot
copies semantics only, never the source pixel buffer. The scene constructor rejects zero or mixed
timebase timing, terminal display or delivery stages, and color history that does not extend the
decoded source history.

`GraphPlaybackFrameEvaluator<T, N, V, D>` uses the existing immutable
`GraphEvaluationSnapshot`, `FrameMemoryCache`, and `OwnedHostFrameMemoryCache`. Its complete cache
scope includes project, media fingerprint, evaluated parameters, scene color, render settings,
graph lineage, exact physical time, output endpoint, and region. A validating cache adapter admits
only scene values whose timestamp, complete color pipeline, and alpha association match the graph
request, so a malformed result cannot poison later reuse.

`PlaybackDisplayTransform<V>` keeps orchestration generic over a future GPU-resident display value.
The concrete `CpuPlaybackDisplayTransform` consumes canonical binary16 `WorkingImage` values,
verifies their scene interpretation, applies the existing `OutputColorTransform`, and produces
premultiplied binary32 display images plus `ViewportColorMetadata`. Native surface acquisition,
queue submission, and presentation remain owned by the viewport and GPU sink.

`PlaybackAudioOutput` wraps the existing all-or-nothing lock-free `OutputProducer` and its paired
`AudioMasterClock`. `PlaybackOrchestrator<O>` accepts borrowed prepared audio, submits one exact
frame without waiting, polls worker completion, waits on the shared clock, and publishes through
the existing bounded graph-to-viewport handoff. It stores only a weak worker-pool reference, never
blocks the playback domain, never invents a frame-drop policy, and retains the exact owned viewport
payload after backpressure.

## Normal, degraded, and recovery behavior

Normal playback evaluates or reuses one exact graph result, performs display conversion on the
worker, advances the audio clock only through the real device consumer, and releases the video
frame when that clock reaches its timestamp. Exact cache reuse skips node execution while returning
the same display semantics.

Audio saturation returns a retryable resource error without accepting a partial submission.
Viewport saturation returns the exact frame to playback for a later poll. A graph result with the
wrong timestamp, scene color, or alpha association returns a degraded corrupt-data failure, is not
retained in the shared cache, and clears only that request. A later valid frame can then evaluate
and present through the same graph, cache, clock, audio, and viewport owners.

Explicit audio-clock loss switches to the monotonic playback clock at the current timeline
position. Recovery switches back to the paired audio clock at the then-current position. Both
operations preserve continuity and leave transport rate, seeking, direction, frame stepping, and
drop policy to their dedicated checkpoint.

## Research basis

GStreamer synchronization and clock design were used as the primary reference for one pipeline
clock, audio-sink clock preference, running-time comparison, and explicit clock reselection.
wgpu 23.0.1 surface documentation confirmed that queue submission precedes presentation and that
presentation remains a sink-side responsibility. CPAL 0.17.3 documentation confirmed that heavy
decode, graph, color, allocation, and locking work must stay outside the high-priority device
callback. The implementation uses only existing pinned MIT or Apache-2.0 dependencies and adds no
network or closed-tier path.

## Verification

The integration contract was written first and failed because all new orchestration surfaces were
absent. Its real consumer path constructs a high-precision decoded frame, typed source metadata,
an immutable graph snapshot, a budgeted cache, a binary16 scene image, an explicit display
transform, the lock-free output buffer and device consumer, an audio master clock, a bounded worker
pool, and a one-slot viewport handoff.

The contract proves playback-domain enforcement, early clock waiting, exact audio-clock release,
decoded precision and straight-alpha provenance, retained media and image metadata, binary32
premultiplied display output, complete color history, exact cache reuse, audio saturation, lossless
viewport backpressure, invalid-scene rejection without cache poisoning, later frame recovery, and
continuity across audio loss and restoration. Focused, package, dependency, map, documentation, and
repository verifier evidence is recorded in the ignored checkpoint execution log.

## Remaining constraints

Source opening, decoder selection, timeline graph construction, prepared audio rendering and
resampling, transport controls, frame-drop policy, A/V drift correction, native GPU display
submission, export, physical display and audio hardware, and long-session soak remain separate
owners or later proof lanes. The CPU display transform is a deterministic concrete consumer, not a
claim that the production viewport is CPU-resident. No public API or application shell invokes this
orchestrator yet.
