# P3.W03.C013 removable volumes and changed sources

## Outcome

The desktop media library now retains exact source baselines, distinguishes a missing mounted file from an offline removable volume, and detects changed bytes without changing stable media identity or silently rebinding editorial state. Editors can check every source efficiently or force exact byte verification for one selected identity, then inspect the required relink intent beside the existing organize, metadata, search, proxy, preview, relink, replace, and conform controls.

## Architecture and integration

`MediaBrowserItem` remains the sole desktop media identity projection. Its persisted `source_monitoring` record stores the authoritative content fingerprint, bounded scan generation, overall status, relink intent, and one ordered path record containing conservative volume identity, accepted fingerprint, current observation, and exact path state. Existing IDs, source metadata, user metadata, annotations, content analysis, selections, bins, proxy and optimized attachments, representation choice, and derived availability remain separate fields and are never rewritten by a scan.

Import computes per-file baselines during the same byte pass that creates the aggregate source fingerprint. This adds no second import read and no field to the serialized import response. Older sidecars load through defaults and establish baselines only after every path is available and the complete aggregate bytes still match the authoritative fingerprint.

The revision-fenced `scan_project_media_sources` command scans selected IDs or the complete current library on a blocking worker. Routine all-source checks compare file size and exact modification time against persisted baselines and avoid reading unchanged large media. Selected exact checks set `verify_content`, hash current bytes even when metadata is unchanged, and compare each path to its accepted baseline. Metadata-only changes refresh the signature, while content changes retain the old baseline and set `review_changed_source`.

Conventional macOS `/Volumes/<name>`, Linux `/media/<user>/<name>`, `/run/media/<user>/<name>`, and `/mnt/<name>` roots are classified conservatively as removable. An absent removable root sets `wait_for_volume`; a missing file on a mounted root sets `locate_source`; unreadable or nonregular paths set `retry_inspection`. Unknown Windows drive roots remain explicit volume identities without an unsupported removable-hardware claim.

## Failure and compatibility behavior

The scan applies to a cloned media-library snapshot and rejects stale revisions, duplicate or excessive IDs, unknown identities, invalid persisted state, or revision exhaustion without partial publication. File disappearance, permission loss, nonregular paths, unmounted volumes, and read races become inspectable per-path state instead of transaction failure or source replacement.

C009 Relink and Replace reset monitoring for the accepted paths and current authoritative fingerprint, while Conform preserves existing source evidence. C012 batch relink reuses that same behavior. A changed source cannot generate a C010 preview under its old fingerprint, and C011 analysis remains retained but never rebound automatically. C008 proxy and optimized choices remain tied to the accepted fingerprint and continue deterministic original fallback.

The implementation adds no watcher, network connection, platform service, hidden retry loop, or automatic replacement. The exact scan command is offline, deterministic, explicitly revision-fenced, and serialized through the existing project media owner.

## Consumer behavior

The strict TypeScript bridge mirrors every monitoring, volume, path, fingerprint, request, and relink-intent field. The production media browser offers `Check all sources` for metadata-efficient project-wide inspection and `Verify selected source bytes` for exact hashing. Cards show overall source status, while the selected inspector shows scan generation, relink intent, volume identity and kind, mount state, accepted and observed fingerprints, size, path state, and actionable detail.

## Verification

The focused `removable_media_contract` freezes the native store, Tauri registration, strict bridge, and real React consumer. Its filesystem tests prove import-time baselines, exact same-path byte-change detection, unchanged identity and editorial state, changed-preview rejection, stale-request rollback, accepted-byte restoration, mounted-volume file loss and return, durable sidecar reload, and absent conventional removable-volume `wait_for_volume` intent. Adjacent import, metadata, derived-media, offline-media, content-search, batch-operation, and generated-preview contracts protect C003 through C012 compatibility.

Rust source-monitoring unit proof checks conservative macOS, Linux, system, and Windows volume identities. Strict TypeScript checking, the production build and frontend suite, Rust formatting and diagnostics, refreshed workspace maps, and the repository checkpoint verifier remain delivery gates.

## Remaining constraints

The metadata-efficient all-source scan cannot detect an adversarial same-size rewrite that also preserves the exact modification timestamp; the selected exact verification path exists for that case. Volume classification intentionally uses stable conventional mount roots instead of claiming hardware enumeration. Native filesystem notification, operating-system device events, automatic background scheduling, and engine-side transcode production remain separate work.
