# P2.W05.C001

## Outcome

Implemented the internal visual effect and node authoring SDK as one graph-native contract. Effect
definitions are typed, editable, deterministic, inspectable, animatable through existing graph
parameter declarations, and reusable without a separate timeline-only state model.

## Architecture and integration

`superi-effects::authoring` composes the existing `superi-graph` contracts instead of duplicating
them. `EffectNodeDefinition<T>` owns one immutable exact `NodeSchema`, inspectable node, port, and
parameter presentation, and typed parameter defaults. Its generic host payload keeps effect state
compatible with the caller's graph value model rather than introducing an effect-only opaque value.

`EffectInstanceBindings` retains caller-owned port and parameter identities. Instantiation applies
validated typed overrides or defaults and returns an ordinary `EditableNode<T>`, so graph
transactions, immutable snapshots, parameter drivers, timeline compilation, scripts, and headless
callers continue to observe the same authored state.

`EffectCatalog<T>` atomically registers richer definitions with the canonical graph `NodeRegistry`.
Its immutable snapshots preserve exact registration revisions and deterministic node-type and
semantic-version ordering even after later catalog changes.

`EffectNodeCompiler<T, N>` binds exact definition versions to `Send + Sync` caller-owned runtime
factories and implements the graph `NodeCompiler` seam. Each factory receives the exact immutable
`GraphSnapshot<T>` and `EditableNode<T>`. Missing definitions, unavailable factories, duplicate
factories, and same-ID schema drift fail with shared classified diagnostics before a runtime graph
is published.

## Implementation process

The public `authoring_contract` was added first and failed on the nine absent SDK surfaces. The
implementation then added validated presentation and defaults, graph-native instantiation, atomic
catalog discovery, and exact-schema runtime factory compilation. Focused review added explicit
coverage for unavailable factories and same-ID schema drift before the verification scope widened.

Primary-source review used OpenFX 1.5.1's immutable descriptor and live-instance split, host-owned
parameter state, and context-specific clip model. The internal SDK adopts the descriptor and
host-owned state principles while deliberately keeping one workflow-neutral graph schema instead
of copying OpenFX contexts or adding an OpenFX dependency. The official ASWF OpenFX project and
reference documentation were the research sources.

## Verification

The public contract constructs one typed image operation with an animatable scalar parameter,
inspects all presentation and graph declarations, registers definitions in reverse order, and
proves canonical immutable catalog and schema snapshots. It instantiates the same definition in
independent timeline-role and node-graph-role editable graphs, applies ordinary parameter mutation
transactions, and observes identical typed state.

The same contract compiles both immutable graph snapshots through one exact factory registration,
proves `Send + Sync` suitability, and verifies atomic duplicate registration, malformed metadata,
mistyped defaults, missing bindings, unknown overrides, unknown and duplicate factories,
unavailable runtime implementations, and same-ID schema drift. Focused package tests,
warnings-denied Clippy, documentation tests, downstream graph, timeline, and engine proof,
formatting, map validation, and the final repository verifier are recorded in the ignored execution
log with their final results.

## Delivery

This record is delivered with the authoring SDK, public consumer contract, crate documentation, and
refreshed codebase maps. The single checkpoint commit is pushed to `origin/main`; its full SHA is
recorded in the live checklist after remote containment is confirmed.

## Remaining constraints

This checkpoint defines authoring, discovery, graph instantiation, and runtime translation seams.
It does not implement built-in transform, composite, blur, keying, or utility nodes, pixel
algorithms, GPU kernels, keyframes, reusable control drivers, masks, roto, transitions, text,
tracking, OFX hosting, persistence, timeline attachment models, rendered frames, or engine playback.
Those remain owned by later effects, graph, timeline, project, and engine checkpoints.
