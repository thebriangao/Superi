# P3.W04.C009 editable timeline transitions

## Outcome

The production timeline now exposes selected transitions as precise editorial objects with exact from and to handles, total duration, start, center, end, or custom alignment, adjacent-media limits, and typed visual parameter controls. Timing publishes through one atomic canonical timeline edit, while graph-owned processing values publish through the existing typed graph mutation path. Both paths refresh the complete application-owned editor snapshot and keep failure state visible without inventing local project state.

## Architecture and integration

`superi-timeline` remains the owner of transition identity, endpoints, overlap timing, item order, and validation. Its new `set_transition` operation replaces both exact offsets inside one unpublished project draft, rejects the wrong track clock, zero duration, no-op edits, insufficient endpoint capacity, and neighboring transition overlap, then reports the transition as the only modified object with unchanged track duration and the union of the old and new visible ranges.

`superi-api` exposes that operation as the additive `set_transition` discriminant under editor schema `1.3.0`. The generated TypeScript client carries the exact timeline, track, transition, from-offset, and to-offset fields, and public command evidence retains the new operation kind. Existing project history continues to own atomic publication, stale-revision rejection, undo, redo, graph reconciliation, and project events.

The desktop application keeps transport ownership in `ApplicationProvider`. `TimelineWorkspace` receives one `executeProjectActions` callback and never calls the generated client directly. Successful and failed edits both trigger a canonical state refresh, so draft input remains transient and every visible published value comes back through the existing editor-state resource.

## Transition projection and gestures

The strict timeline canvas projection now retains each transition's exact offsets. A separate immutable transition projector joins those offsets with track adjacency and the one graph document owned by the root timeline. It derives exact duration, deterministic alignment, maximum handles after accounting for a transition on either opposite clip edge, stable endpoint and graph identities, downstream processing nodes, schema animatability, parameter drivers, and host-owned restrictions.

Selecting exactly one transition reveals a dedicated inspector. Editors can enter exact handle values, enter a total duration, apply the current timing, or choose start, center, or end alignment without changing total duration. Impossible alignments are disabled, odd centered durations use a deterministic lower from handle and remaining to handle, and unchanged timing is rejected before transport. The transition item itself also shows its exact handle pair on the canvas.

Scalar, boolean, and choice parameters use typed controls and the existing `mutate_graph` action. Canonical scalar bit patterns are decoded to finite display values, directional wipe choices use the built-in four-value catalog, and outbound values use the public `EditorGraphValue` union. Driven values, transition progress, unknown value kinds, and unsupported parameters remain inspectable but noneditable.

## Failure handling

Strict timing projection rejects invalid adjacency, clocks, zero duration, overlong handles, and conflicting opposite-edge consumption. Missing or malformed optional graph detail leaves exact timing available and reports parameter state as unavailable. The command owner classifies transport and public failures, keeps the last valid snapshot, refreshes canonical state, and displays the corrective result beside the relevant transition or parameter control.

## Verification

The lower timeline contract proves exact dual-handle publication, affected-range evidence, unchanged duration, no-op rejection, wrong-clock rejection, capacity validation, and atomic rollback. The public editor contract proves the new strict discriminant, schema `1.3.0`, real command publication, returned evidence, event correlation, and undo through `ProjectEditorApi`; the public schema and generated-binding contracts cover the versioned wire boundary.

The frontend transition contract proves handle limits across neighboring transitions, exact alignment and duration derivation, canonical scalar decoding, graph traversal, driver and host restrictions, strict command payloads, immutable output, no-op prevention, and timing survival when graph detail is malformed. The complete frontend suite, TypeScript check, production Vite build, focused Rust suites, codebase-map validator, and repository checkpoint verifier cover the production `EditingWorkspacePanel` consumer and every changed owner.

## Remaining constraints

This checkpoint exposes the processing value kinds used by built-in transitions: scalar, boolean, and choice. Vector, color, matrix, arbitrary choice catalogs, driver authoring, effect attachment, and transition creation or removal remain future graph and editorial tools. Timeline values stay exact strings in the presentation layer, but the current public number wire still limits newly authored handle values to JavaScript's safe integer range.
