# P3.W06.C012

## Outcome

Implemented one fixed, versioned color-critical dark environment for the production desktop
application. Semantic theme tokens now control shared application chrome, status, focus, selection,
and overlay presentation, while separately named viewer and marker color-data tokens preserve their
existing meaning. Static HTML declares the contract before JavaScript and deterministic bootstrap
activation repairs drift without reading or mutating project, workspace, transport, or persisted
application state.

## Architecture and integration

`app/src/theme.ts` owns the sole supported theme identity, schema, browser color, scene appearance
owner, and workspace-state policy. `applyApplicationTheme` reconciles only three root attributes and
the browser theme-color metadata, returns frozen repair evidence, and publishes ready or recovered
status on the document root. It has no storage, Tauri, generated API, project-action, viewer-state,
or network dependency.

`app/src/theme.css` is the value authority for application surfaces, borders, text, focus,
selection, status, and shadows. The production HTML declares `color-critical-dark` before startup,
and `main.tsx` activates it before constructing the desktop transport or rendering React. Shared
shell controls, the complete command palette, configurable shortcut editor, tooltips, context menus,
notifications, progress, status, and classified recovery surfaces consume semantic `--theme-*`
tokens.

Viewer surrounds, analysis overlays, comparison presentation, and authored marker flags use
separate `--viewer-*` and `--marker-*` color-data namespaces with their prior exact values. The
native viewer reservation is isolated from forced-color substitution and explicitly retains full
opacity, normal blending, and no CSS filter. Native frame pixels remain outside the webview, while
monitor profile identity, display intent, canonical ACEScg meaning, RGBA16F precision, and the
six-stage display transform remain owned by the existing native GPU and `superi-color` path.

## Implementation process

The focused theme test first failed for the intentionally absent `app/src/theme.ts`. Implementation
then added the immutable contract, deterministic repair, static declaration, pre-application
activation, token authority, shared chrome adoption, complete palette adoption, native viewer
isolation, and exact semantic color-data aliases without introducing a second theme, system color
preference branch, persistence owner, or project command. Final integration retained the later
shortcut and application-feedback behavior while moving their chrome and all four recovery classes
onto the same theme contract.

## Remaining constraints

The open application intentionally exposes only the color-critical dark theme. Adding another
theme requires a new product decision, exact token review, migration and recovery behavior, and
proof that scene pixels, analysis meaning, authored marker meaning, monitor binding, and display
transform order remain unchanged. Arbitrary ICC matrix, TRC, and LUT execution remains outside this
checkpoint and stays explicitly owned by the existing color pipeline roadmap.
