# P2.W07.C016

## Outcome

Implemented one stable public command surface for the complete currently executable authored
project operation set. `ProjectEditorApi` accepts strict schema `1.0.0` commands at
`superi.project.command.execute`, routes them through the authoritative engine dispatcher, and
returns typed project history state plus semantic action evidence. Authored changes publish the
correlated `superi.project.state.changed` event and expose `superi.project.history` as the minimum
replacement resource needed before the complete state snapshot checkpoint.

The surface includes apply, undo, redo, and inspect. Apply accepts one ordered compound transaction
containing root timeline selection, timeline editing, graph mutation, referenced-media mutation,
clip mix mutation, and durable extension mutation. The public vocabulary covers all 15 timeline
operations, all 8 graph operations, all 3 media operations, all 4 clip mix operations, and all 6
extension operations supported by the production project transaction owner.

## Boundary and ownership

Wire-visible DTOs belong to `superi-api`. They use explicit snake-case discriminators, strict
unknown-field rejection, canonical typed identifier parsing, exact rational time, checked finite
numeric conversion, complete material and graph value descriptions, declared media path syntax,
and complete extension state. Lower crates do not derive or own the public wire schema.

`superi-engine::editor` is a curated construction seam over the already checked lower-domain types.
It adds no state, mutation algorithm, history, dispatcher, persistence owner, or wire contract.
This preserves the reviewed runtime dependency direction from `superi-api` to `superi-core` and
`superi-engine` without adding direct API dependencies on timeline, graph, project, or audio
crates.

Every public apply is fully converted before dispatch and becomes exactly one
`CompoundProjectTransaction`. The engine dispatcher remains the only sequencing and event owner,
and `ProjectCommandHistory` remains the only undo and redo owner. A mixed transaction is therefore
one project revision, one history unit, and one correlated event, never a sequence of partially
published subsystem changes.

Existing settings, audio automation, recovery, diagnostics, scenario, JSON-RPC, error, and
capability contracts keep their permanent names and behavior. Playback transport, host lifecycle,
work permits, prepared exports, event subscriptions, complete project snapshots, permissions,
bindings, scripting runtimes, CLI execution, command logs, and extension registration policy stay
with their later checkpoints.

## Failure, durability, and automation behavior

Malformed identifiers, invalid timebases, invalid ranges, non-finite controls, invalid paths,
invalid graph values, invalid extension records, empty batches, and oversized collections fail
through checked conversion or the existing domain validation. Conversion completes before any
dispatcher call, so these failures do not advance project revision, command sequence, history, or
event state. Stale revisions, late domain failures, and event backpressure retain the engine's
existing all-or-nothing behavior.

Successful results preserve the caller transaction ID, engine command sequence, public command
kind, authored-change flag, history depths, next undo and redo kinds, and ordered semantic action
evidence. Events preserve event sequence, command sequence, transaction ID, project revision,
replacement history state, and the same evidence. Inspect is revision fenced and eventless, while
undo and redo publish fresh monotonic project revisions when they change authored state.

The public facade exposes its immutable selected project snapshot to the existing in-process
persistence consumer. The contract proof persists a mixed public transaction through
`ProjectDatabase`, reloads an identical project snapshot, then undoes and redoes the transaction
through the same public command surface. Crash recovery remains the existing project recovery API
over the same durable aggregate rather than a second editor-specific store.

## Verification

The strict editor contract fixes every permanent method, event, resource, command, action, and
operation discriminator. It rejects unknown fields and unsupported discriminators, proves malformed
ID, timebase, and non-finite conversion atomicity, and exercises a real mixed graph, timeline,
media, clip mix, extension, and root-selection transaction through EngineControl.

The mixed fixture verifies one revision, one undo unit, ordered six-action evidence, one correlated
event, exact authored timeline and media changes, retained clip controls and extension state, an
identical database reload, and fresh undo and redo revisions. Existing package contracts verify
that schema discovery, settings, automation, recovery, diagnostics, and scenario compatibility are
unchanged. Warnings-denied Clippy, documentation, dependency policy, maps, and the repository final
verifier complete the checkpoint proof.

## Remaining constraints

The current event drain is intentionally scoped to a dispatcher owned by `ProjectEditorApi`; it
does not consume unrelated subsystem events. Reconnectable multi-subscriber delivery belongs to
C019. The history resource is intentionally minimal because complete stable project state snapshots
belong to C017. Later scripting, SDK, CLI, permissions, negotiation, logging, and extension policy
layers must consume this command vocabulary instead of defining competing editor mutations.
