# P2.W02.C013

## Outcome

Implemented faithful OTIO 0.18.1 JSON import and export over the ordinary Rust-native editorial
model. Editors can import the pinned canonical and coverage timelines, reshape every supported
track object through `EditorialProject`, and export the current state without losing preserved
metadata, extensions, naming, color, audio, quality, resumability, or unsupported effects.

## Architecture and integration

`superi-timeline::otio` owns the interchange boundary. `import_otio` parses JSON without Python,
plugins, network access, or a closed runtime, assigns deterministic typed native identities, and
returns an `OtioDocument` that owns an ordinary `EditorialProject`, its root `TimelineId`, stable
diagnostics, and opaque source templates. `OtioDocument::project_mut` exposes the existing atomic
`EditorialProject::edit` transaction rather than a parallel timeline model.

The supported native subset maps `Timeline.1`, `Stack.1`, `Track.1`, `Clip.2`, `Gap.1`,
`Transition.1`, `Marker.2`, `ExternalReference.1`, and `LinearTimeWarp.1`. Root and nested stacks
become native timelines; a nested stack also becomes a parent clip with a typed child-timeline
source. Track order defines record placement, transitions retain exact adjacency and handles,
markers retain timeline, track, or object ownership, and linear time warps become clip-owned exact
playback rates.

OTIO time values are converted only when they are exactly representable on the target native
clock. Common fractional video rates retain their exact rational `Timebase`, generic audio tracks
receive explicit import-policy sample rate, layout, and routing, and no binary floating-point
rounding is allowed to alter a record boundary.

Each supported object retains its complete source JSON template beside the typed native identity.
Export rebuilds hierarchy and ordering from the current project, patches supported names, ranges,
media links, transition handles, marker values, metadata, and linear retime scalars, then merges
those values into the template. Unknown fields, inactive media references, format extensions, and
unsupported effects therefore remain intact. New native objects receive canonical OTIO shapes and
stable Superi identity metadata.

`FreezeFrame.1`, unsupported `Effect.1` objects, and unknown timed children remain preserved. Each
unsupported construct produces `timeline.otio.unsupported_construct` with warning severity, the
exact source JSON pointer, source object ID when available, and OTIO schema. Unknown timed children
with a usable range remain directly visible as native generators while their original schema stays
authoritative for export.

`OtioSchemaTarget::OtioCore0181` exposes the complete schema-version map reported by the exact
OpenTimelineIO 0.18.1 release. The only new production dependency is the workspace-pinned
`serde_json`; the Apache-2.0 OpenTimelineIO package remains an external verification oracle and is
not linked, vendored, or required at runtime.

## Public consumer

The `otio_roundtrip` example is a real headless consumer:

```text
cargo run -p superi-timeline --example otio_roundtrip -- <input.otio> <output.otio>
```

It imports through the public API, reports stable warnings, and writes deterministic OTIO 0.18.1
JSON through the same export boundary used by other callers.

## Implementation process

The production contract was added first and failed because `otio.rs` exposed no reader, writer,
target, options, or diagnostic API. The implementation then made the canonical fixture import into
one 48-frame native timeline and the coverage fixture import into a 120-frame root plus a 48-frame
nested timeline. Further red contracts covered duplicate source identities, exact-clock rejection,
explicit audio defaults, native edit reimport, deterministic bytes, unsupported preservation, and
native linear retime replacement.

The official reference package initially failed under Python 3.14 while loading its compiled
manifest. A clean Python 3.12 environment with exactly `opentimelineio==0.18.1` loaded, target-wrote,
and reread both Rust-produced outputs successfully.

## Verification

The new nine-test interchange contract proves native fixture structure, exact 48-frame and
120-frame duration, media links, transition adjacency, nested ownership, two linear playback rates,
two required warning pointers, deterministic export, opaque effect retention, direct native name
and retime edits, reimport, explicit 48 kHz stereo audio policy, duplicate identity rejection, and
inexact timing rejection with the originating JSON pointer, and conflicting repeated media
identity rejection, and a real native sequence reshape that removes transition and gap objects,
reflows the nested link, rebuilds OTIO hierarchy, and reimports at the new exact duration.

The complete `superi-timeline` suite passes all 62 tests across model, edit state, markers,
foundational and advanced edit operations, nesting, ranges, retiming, fixture structure, and
production interchange. Warnings-denied Clippy passes for every timeline target, and formatting is
clean.

The public example produced `/tmp/p2-w02-c013-rust-canonical.otio` and
`/tmp/p2-w02-c013-rust-coverage.otio`. OpenTimelineIO 0.18.1 under Python 3.12 deserialized both,
reported one track and durations 48 at rate 24 and 120 at rate 24, target-wrote both with its exact
0.18.1 schema map, reread them, and returned `is_equivalent_to == true` for each.

Workspace tests, dependency direction, the offline boundary scan, map validation, and final remote
integration remain delivery gates and are recorded in the ignored execution log after they run.

## Remaining constraints

The native subset does not interpret arbitrary vendor effects or every future OTIO schema. Such
objects remain opaque and diagnosed rather than dropped or guessed. A generic OTIO audio track
does not itself identify sample rate, channel layout, or routing, so those values remain explicit
import policy while original audio metadata stays preserved. Variable piecewise native time maps
remain represented in the native model, but only a one-segment rate has a standard
`LinearTimeWarp.1` projection in this checkpoint. Multicam, timeline-to-graph compilation,
persistent project serialization, and editor or API exposure remain separately owned work.
