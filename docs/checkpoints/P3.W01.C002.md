# P3.W01.C002

## Outcome

Linked the production Tauri process to one long-lived, lifecycle-attached headless engine on its
dedicated EngineControl thread and exposed the existing integration-validation result through a
bounded transport-neutral Rust connection.

## Architecture and integration

`LinkedEngineProcess` is the explicit native owner of one managed EngineControl thread, while
`EngineConnection` is the cloneable handle retained by the Tauri shell. The connection admits only
the existing typed integration-validation query without waiting for queue capacity, and
`PendingEngineValidation` moves response waiting to a blocking-safe caller. No Tauri API command,
JSON-RPC router, frontend transport policy, reconnection, cancellation, or transport-specific error
surface is added by this checkpoint.

The EngineControl owner constructs and retains one full `EngineCommandDispatcher` for each exact
application engine generation. It consumes C001's `HeadlessEngineLifecycleParticipant` only through
its signal, acknowledgement, and failure operations, begins the real dispatcher shutdown before
acknowledging application stop, and constructs a fresh dispatcher after restart. Validation flows
through `EngineCommandDispatcher::integration_validation_snapshot`, the live media declaration
registry, `IntegrationValidationApi`, and `GetEngineIntegrationValidationResult`, so the application
observes the same canonical engine state as scripts and headless tools without fabricating subsystem
readiness.

## Implementation process

The synchronized base was `3b19467ebe48877f0b5612cb1116e8542523c935`, containing the complete C001
lifecycle shell. The focused native contract first failed because the application had neither a
`superi-api` dependency nor an engine connection module. The smallest implementation then added the
two downward engine/API path dependencies, one fixed-capacity native request queue, one
EngineControl owner loop, exact lifecycle token handling, and the production bootstrap link. No
generated TypeScript or command/event transport surface changed.

## Verification

The focused engine connection contract passes three cases for exact managed-thread ownership,
truthful public validation projection, bounded nonblocking admission, stable connection reuse across
a fresh application generation, and orderly stop and join. All native application unit, lifecycle,
connection, and documentation tests pass from the locked standalone manifest. The existing public
API and engine integration-validation suites each pass three compatibility cases, and the production
frontend passes strict type checking, Vite build, and all four application contracts.
