# P2.W04.C011 Audio automation

## Outcome

Implemented exact clip-gain keyframes plus Read, Write, Touch, and Latch automation modes. Ordered
revision-fenced transactions preserve untouched curve regions, publish complete replacement state,
and drive audible gain from exact absolute sample coordinates through the existing routed mix graph.

## Architecture and integration

`superi-audio::automation` owns typed clip-gain targets, bounded authored lanes, professional mode
state, half-open write regions, immutable replacement snapshots, and prepared curves. Transactions
apply to a cloned candidate and publish atomically only when semantic state changes. Prepared curves
replace fixed clip gain explicitly through `ClipMixSnapshot::prepare_processor_with_automation`;
the existing preparation method and every routing, pan, phase, fade, mute, and solo behavior remain
unchanged when no automation snapshot is supplied.

The callback consumes only immutable prepared points. Per-frame evaluation uses the absolute signed
sample coordinate with binary search and linear interpolation, performs no allocation, locking,
dispatch, or state mutation, and therefore produces identical samples across whole and irregularly
partitioned blocks.

`superi-engine::dispatcher` owns lifecycle-scoped attachment, serialized inspection and mutation,
event-capacity reservation, command ordering, no-op suppression, and isolated automation revision
correlation. `superi-api::audio_automation` projects that owner through strict schema 1.0.0 JSON,
canonical clip identifiers, permanent method and event names, typed transactions, and complete
replacement events without adding a direct API dependency on the audio crate.

## Implementation process

Public audio, engine, and API contracts were each added and observed failing on their intended
missing boundary before implementation. The audio state machine then established exact Read,
Write, Touch, and Latch behavior; the dispatcher integrated authoritative ownership and atomic event
publication; and the API added one-to-one strict wire conversions. Review added explicit rejection
for writable `i64::MAX` coordinates, where no exclusive region end can be represented, and changed
capacity validation to evaluate the completed ordered candidate so valid replacement transactions
are not rejected conservatively.

## Verification

Focused audio tests prove direct and interpolated keyframes, negative coordinates, semantic no-ops,
wrong clocks, stale and invalid rollback, exact Write replacement, multiple Touch regions, Latch
hold, coordinate overflow, and split-block equivalence. A real stereo graph renders a constant
source through the automated clip processor, submix, and master with exact expected nonzero samples
and unchanged channel meaning.

Focused engine tests prove exclusive full-dispatcher ownership, inspection ordering, one correlated
event per semantic transaction, stale and invalid atomicity, a full 64-event backpressure boundary,
and consumption of the returned snapshot by the real clip processor. Focused API tests prove every
mutation projection, strict unknown-field rejection, canonical typed IDs, schema and permanent
names, complete result and event JSON round trips, Touch behavior, no-op suppression, and conversion
failure atomicity. Complete audio, engine, and API package suites, formatting, warnings-denied
Clippy, rustdoc, dependency direction, codebase maps, and the repository verifier form the final
delivery proof.

## Delivery

This record ships with the audio state machine and prepared consumer, serialized engine boundary,
strict public API, integration contracts, and refreshed codebase maps. The delivered commit is
pushed to `origin/main`, and its full SHA is recorded in the paired checklist after remote
containment is confirmed.

## Remaining constraints

Schema 1 automates clip linear gain only. Durable project serialization, pan, mute, send, bus,
effect, and plugin parameter targets, touch hardware ingestion, physical device latency, and
hardware soak evidence remain owned by later checkpoints. Existing device, export, scheduling,
metering, routing, and plugin systems are preserved rather than widened by this checkpoint.
