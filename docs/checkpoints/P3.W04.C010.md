# P3.W04.C010 durable timeline markers

## Outcome

The editing workspace now creates, displays, navigates, edits, removes, and immediately reverses durable timeline markers through one revision-fenced project command path. Labels, flags, notes, exact owner-clock ranges, owners, metadata, and non-navigable state remain visible and dependable without introducing a second frontend mutation owner.

## Architecture and integration

`superi-timeline` owns one nonempty ordered `MarkerMutation` batch for create, range, label, flag, note, and remove operations. The complete create operation retains owner, exact range, optional visible fields, and recursive metadata, while partial updates preserve every unmentioned field. One project edit closure validates and applies the entire batch, so a duplicate identity, missing target, invalid field, or later failure leaves canonical state and revision unchanged.

`superi-engine` publishes marker batches through the existing compound project transaction. The action records typed `markers_mutated` evidence, recompiles retained timeline graphs through the established edit-preservation path, and stores one complete history snapshot. Long-lived engine consumers can undo and redo the full marker state without losing compiled edit relationships.

`superi-api` exposes strict tagged marker mutations and exact marker owner, flag, range, metadata, and evidence types through the unchanged `superi.project.command.execute` method. The additive public catalog is 1.6.0 and the project-editor schema is 1.4.0. The selected-project local host accepts marker batches, persists every successful revision, and reloads canonical state for the next desktop command. The generated TypeScript artifact derives from the Rust contract, while the native dispatcher continues to use its generic project-command route.

The editing workspace reuses `ApplicationProvider` for unique transaction identity, revision fencing, persistence, event delivery, and replacement snapshot refresh. `TimelineWorkspace` receives only a typed marker callback. Its compact marker panel shows every authored marker, seeks exact targets, navigates previous and next exact markers, creates at the exact playhead, atomically saves range and visible fields, removes markers, reports pending or failed commands, and sends the complete typed inverse of the latest successful gesture for immediate reversal.

## Exactness and visible state

Timeline and track markers retain their exact owner clock. Object markers retain their exact object-relative range and resolve against the object's canonical record start. Exact marker targets participate in stable chronological navigation and existing snap rules, with marker identity resolving ties. Markers whose coordinates cannot be rescaled exactly, or valid object markers outside their owner's resolved duration, remain visible and editable with an explicit `Not navigable` state but never become rounded snap or navigation targets.

Removal reversal recreates the complete marker, including owner and metadata. Edit reversal restores the previous range, label, flag, and note in one atomic batch. Creation reversal removes the new identity. Each inverse is usable only at the exact refreshed project revision produced by its gesture, so an intervening authored revision cannot apply stale reversal intent.

## Failure and compatibility behavior

Empty batches, duplicate identities, missing timelines or markers, invalid identities, malformed owner references, invalid exact clocks or ranges, and rejected visible fields fail before publication. A failure at any batch position preserves project state, revision, engine history, graph state, and durable storage. Existing marker serialization, snap behavior, public method identity, event identity, project format, timeline primitive schema, track controls, selection, and application transport ownership remain unchanged.

## Remaining constraints

The desktop host intentionally reloads durable project state between commands, so its in-memory engine undo stack is not a cross-request authority. Immediate desktop reversal therefore uses the complete revision-fenced typed inverse marker batch, while long-lived engine clients retain ordinary history undo and redo. Marker editing currently exposes timeline-owned creation and edits every existing owner kind without moving marker ownership; a future ownership-transfer operation can extend the same atomic boundary if product requirements call for it.
