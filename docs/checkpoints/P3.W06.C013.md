# P3.W06.C013

## Outcome

Implemented one global, action-specific undo and redo presentation backed by the existing engine
transaction history. The always-visible workspace controls, searchable command palette, and native
Edit menu now name the exact next transaction while retaining stable shortcuts, current counts,
document identity, session scope, disabled reasons, and safe-close visibility.

## Architecture and integration

`ProjectCommandHistory` remains the only transaction owner, and the generated editor replacement
state remains the only frontend source for `undo_depth`, `redo_depth`, `next_undo`, and `next_redo`.
The pure `project-history.ts` presentation model combines that public state with the active desktop
document identity and busy state. It requires exact project and revision agreement, validates depth
and next-action coherence, normalizes every generated mutation kind, maps future values to the safe
generic `unknown` category, and fails closed while state is synchronizing.

`ProjectHistoryControls` is always mounted in the shared workspace header. It exposes the active
document name, revision-aware session status, exact action titles, transaction counts, accessible
tooltips, and disabled state on every application route. Both buttons call the existing
`executeHistory` callback, which rechecks action availability, project identity, and revision before
entering `ApplicationProvider.executeProjectCommand`. No React history, project mutation, transport,
or persistence owner was added.

The command palette consumes the same frozen action presentation instead of independently deriving
generic history labels. The private desktop-shell synchronization now carries typed next undo and
redo mutation categories beside the existing depths. Rust validates that no inactive document can
carry history and that zero and nonzero depths agree exactly with the corresponding next action.
Native Edit menu labels use the validated snapshot while preserving `CmdOrCtrl+Z` and
`CmdOrCtrl+Shift+Z`.

Workspace state, keyboard shortcuts, recent projects, and document selection retain their existing
owners. History depths and action kinds remain process-local and are excluded from the schema 3
desktop persistence record. Safe closing continues to block active work, save through the durable
project owner, and warn before any last-known nonempty session history ends.
