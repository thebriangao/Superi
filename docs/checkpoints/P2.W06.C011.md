# P2.W06.C011

## Outcome

Implemented engine-owned OpenFX plugin discovery, isolated loading coordination, descriptor and
schema validation, lifecycle supervision, failure containment, recovery, and quarantine. Playback,
rendering, and export now resolve exact authored graphs against one immutable active plugin registry
revision, so a disabled or failed plugin becomes an ordinary missing-node condition without changing
editable state.

Discovery recursively traverses explicit local roots in deterministic order, validates the standard
`NAME.ofx.bundle/Contents` package shape, skips names beginning with `@`, refuses symbolic-link
bundles, deduplicates overlapping roots, and retains malformed or inaccessible paths as classified
failures. The OpenFX environment search path and platform default root are exposed separately so
applications can inspect or amend policy before scanning.

## Architecture and integration

`PluginSupervisor` owns accepted plugins as heterogeneous boxed `IsolatedOfxAdapter` hosts. A small
forwarding implementation in `superi-effects` preserves the existing safe `OfxPluginHost` contract
while allowing one engine owner to retain adapters from different platform launchers. OpenFX scan
ordering, descriptor validation, graph-native definition projection, instance validation, action
deadlines, fault recording, restart, and quarantine remain effects-owned instead of being copied
into the engine.

`OfxWorkerLauncher` is the platform boundary. Its contract requires architecture-specific binary
selection and validation, target operating-system containment, default-deny access, and versioned
bounded IPC. Portable engine code never maps native plugin code into the editor process and never
offers an in-process fallback. The returned adapter is still rejected unless the effects host
observes the required worker-process isolation, protocol revision, message bound, render deadline,
and restart support.

Each successful scan projects immutable schemas into a discovered registry. The supervisor validates
every candidate against a cloned aggregate registry before acceptance, so duplicate exact identities
or cross-plugin schema conflicts reject only the offending bundle. The active registry is rebuilt
from the ready subset after every lifecycle or instance action, providing real removal semantics for
disabled, faulted, and quarantined plugins without mutating the graph or inventing a second effect
catalog.

## Permissions and failure behavior

Scanning always uses the permission-free OpenFX load request. Activation accepts a caller-owned
authorization set, proves that every requested capability is authorized, then forwards only the
plugin's exact requested set. Unrelated authorized capabilities never reach the worker.

Discovery, launch, validation, and latest worker failures are retained as `PluginFailure` snapshots
with stage, source, shared category, exact recoverability, concise message, and structured contexts.
Their recommended actions map directly from the shared vocabulary: retryable work retries or
restarts, degraded work continues without the unavailable result, user-correctable work requests a
configuration or permission change, and terminal work stops the unsafe operation.

One broken bundle cannot abort unrelated discovery or scan. A worker action failure updates only its
own host, immediately rebuilds active graph availability, and preserves its actionable OpenFX action
record. Repeated failures cross the configured quarantine threshold, clear live instance handles,
and require explicit acknowledgement before restart. Total and latest failure evidence remain
visible after recovery.

## Playback, rendering, and export coherence

`PluginWorkResolution` consumes the existing `EngineWorkKind` vocabulary rather than defining a
parallel workflow model. All three work kinds resolve through the same supervisor revision and the
same active `NodeRegistrySnapshot`. Exact ready schemas evaluate normally; disabled, faulted, or
quarantined schemas produce the graph-owned degraded missing-node error while authored definitions,
ports, parameters, drivers, edges, and document bytes remain unchanged.

Terminal failure evidence from an unrelated bundle remains inspectable but does not block a graph
whose required plugins are ready. Conversely, no work kind receives a private fallback registry, so
playback cannot silently use a plugin that rendering or export considers unavailable.

## Research basis

OpenFX 1.5.1 packaging defines recursive search, the `NAME.ofx.bundle/Contents` hierarchy, ordered
`OFX_PLUGIN_PATH` roots, platform defaults, and mandatory exclusion of names beginning with `@`:
<https://openfx.readthedocs.io/en/main/Reference/ofxPackaging.html>.

The OpenFX core lifecycle defines binary loading, plugin enumeration, API and version checking, and
host registration before plugin use:
<https://openfx.readthedocs.io/en/main/Reference/ofxCoreAPI.html>.

Apple identifies XPC as the preferred macOS privilege-separation boundary and distinguishes it from
ordinary child sandbox inheritance:
<https://developer.apple.com/library/archive/documentation/Miscellaneous/Reference/EntitlementKeyReference/Chapters/EnablingAppSandbox.html>.
Microsoft documents AppContainer default-deny process, filesystem, network, capability, and job
controls:
<https://learn.microsoft.com/en-us/windows/win32/secauthz/createprocessinsandbox>.
Linux kernel documentation states that seccomp reduces exposed syscall surface but is not a complete
sandbox, and documents `no_new_privs` as a required building block:
<https://www.kernel.org/doc/html/v5.9/userspace-api/seccomp_filter.html> and
<https://kernel.org/doc/html/v5.10/userspace-api/no_new_privs.html>.

These sources support the selected portable coordinator and platform launcher boundary. They do not
justify claiming that one generic Rust process launcher provides equivalent containment on every
operating system.

## Verification

The public engine contract was written first and failed only because the planned discovery,
launcher, failure, and supervisor surfaces did not exist. Its completed fixture launches a real
child test process for each bounded adapter action, validates child exit and protocol output, and
proves permission-free scan, activation, deadline failure, restart, render fault, and quarantine
through the dynamic adapter path used by the production supervisor.

The same contract creates nested healthy, broken, malformed, and `@`-ignored bundle packages. It
proves deterministic discovery, malformed-package retention, contained terminal launcher rejection,
healthy plugin acceptance, exact permission narrowing from a broader authorization set, retryable
worker evidence with process context, two-failure quarantine, explicit recovery, and identical
playback, rendering, and export revisions in both degraded and ready states.

Focused engine and effects contracts, complete package suites, the graph missing-node boundary,
warnings-denied Clippy, documentation tests, formatting, map validation, and the repository final
verifier form the local completion evidence. Exact commands and results remain in the ignored
checkpoint execution log.

## Remaining constraints

Concrete XPC, AppContainer, and Linux namespace, LSM, and seccomp launchers remain platform-owned.
This checkpoint defines and exercises the engine coordination boundary but does not claim a portable
generic sandbox, native OFX ABI bridge, signature scanner, GPU-handle transport, plugin UI panels,
WASM runtime, audio plugin host, or binary cache.

The current OpenFX effects host models one plugin identity per adapter. Platform launchers that find
multiple OpenFX entries in one binary must expose one isolated adapter per exact identity before
passing them to this supervisor. Hardware, hostile-plugin, and long-running soak evidence remains
required for release claims beyond the deterministic headless contract.
