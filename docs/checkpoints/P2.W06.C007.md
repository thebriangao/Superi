# P2.W06.C007

## Outcome

Implemented the engine-wide typed command and transaction dispatcher. Supported scenario edits,
lifecycle changes, runtime failure reporting, recovery, shutdown, restart, workflow admission,
permit validation, and exact interactive transport controls now enter one engine-owned command
boundary with stable transaction identity, monotonic successful-command sequencing, classified
failures, exact recovery automation, bounded logical export-job control, and ordered full
replacement state events.

Canonical scenario automation now accepts an optimistic ordered transaction instead of relying on
uncoordinated one-action calls. Every mutating transaction either publishes all actions as one
revision and one undo unit or leaves scenario state, history, command sequencing, and the event
stream unchanged. The original one-action public command remains compatible by wrapping the same
transaction path.

## Architecture and integration

`EngineCommandDispatcher` owns the canonical `ScenarioEngine` and, in its coherent engine form, one
`EngineLifecycle` plus its canonical `EngineErrorCoordinator`. `EngineCommandRequest` pairs one
bounded caller-owned `EngineTransactionId` with an `EngineCommand`. The command vocabulary covers
scenario execution and inspection, lifecycle inspection and action completion or failure,
operation-labeled classified failure reporting, exact recovery start, completion, failure and
inspection, shutdown, restart, playback or rendering or export admission, permit freshness checks,
and every current interactive transport control. It also covers prepared export submit, automated
polling, inspection, pause, resume, retry, cancellation, and finalized-job removal.
An all-job cancellation command settles unfinished work before blocking-safe worker shutdown.

The lifecycle-attached dispatcher requires the `EngineControl` execution domain for construction,
inspection, mutation, and event draining. This keeps every lifecycle transition and work-admission
decision behind the existing authoritative owner. A deliberately narrower `scenario_only`
constructor preserves the transport-neutral canonical reference surface without claiming control
of a production application lifecycle.

Successful commands receive one monotonic command sequence. State-changing commands also enqueue
one bounded `EngineEventEnvelope` with its own monotonic event sequence, the originating command
sequence, exact transaction identity, optional scenario revision, and complete replacement scenario
or lifecycle state. Classified failure and recovery commands emit one independently revisioned
replacement state containing the coherent lifecycle snapshot, every active failure in subsystem
order, bounded diagnostic history, and the exact pending recovery token. Playback commands reserve
the same event capacity and sequence before crossing to the playback owner, then return complete
replacement transport state after success or failure. Export commands return the complete retained
queue and emit its dispatcher-owned revision after each explicit or newly observed automated
transition.
Capacity and sequence exhaustion are checked before mutation, so backpressure cannot produce an
unpublished state change. Scenario and lifecycle inspection plus permit validation do not invent
state events.

## Atomic scenario transactions

`ScenarioTransaction` carries one expected revision and one to 64 ordered actions. Empty inputs,
oversized inputs, and mixed history actions fail before mutation. `ScenarioEngine` stages ordinary
actions against private content, assigns one resulting revision to their operation evidence, and
publishes one before and after history entry only after every action succeeds. A failing later
action therefore rolls back prior staged work without compensating mutations.

Undo and redo remain one-action transactions because each is already a history operation. One undo
reverses the entire prior multi-action transaction and one redo reapplies it. Optimistic revision
fences reject stale clients before execution and return the complete last valid public state.

Transaction identifiers are nonempty, limited to 128 bytes, and exclude control characters.
Classified failure operation labels are nonempty and limited to 128 bytes. The pending event queue
is limited to 64 envelopes, retained failure context is limited to 16 frames, and copied failure
messages are limited to 1,024 bytes. These limits keep transaction input, pending state publication,
failure message size, and retained context count explicit.

## Lifecycle coherence

Playback, rendering, and export admission are routed through the same lifecycle snapshot used by
direct lifecycle callers. Healthy operation grants revision and lifetime scoped permits. A
recoverable playback failure denies playback while rendering and export remain available, and a
rendering failure also denies export because export depends on rendering. Recovery clears the
active blocker, while stale permits remain invalid after any lifecycle change.

The dispatcher never calls the lifecycle failure and recovery methods as a parallel path. Both the
new classified commands and the legacy runtime-failure and begin-recovery commands route through
`EngineErrorCoordinator`. A legacy lifecycle completion or failure token that addresses an active
recovery also resolves through the coordinator. Legacy callers retain their lifecycle result shape,
but classification-specific intent, exact sequence and attempt identity, failed-recovery
reclassification, diagnostic retention, and stale-token rejection remain mandatory.

Terminal failure, orderly shutdown, and restart use the existing dependency-aware lifecycle
actions rather than bypassing subsystem ownership. Restart completes teardown, begins a fresh
lifetime, and requires the normal initialization sequence before work is admitted again. The
dispatcher reports these changes as full lifecycle replacement events in exact commit order. When
the logical export queue is attached, an export teardown token cannot complete until all retained
jobs are final; automation can issue all-job cancellation and poll acknowledgements before
completing that lifecycle action.

## Classified failure and recovery state

`EngineFailureOperation` validates the stable bounded label supplied by a subsystem owner.
`ReportClassifiedFailure` converts bounded cloneable failure evidence back into the core error
contract and lets `EngineErrorCoordinator` derive the permanent retry, continue-degraded,
user-correction, or restart disposition. The coordinator remains the only owner of active failure
records and diagnostic history, while `EngineLifecycle` remains the only owner of subsystem
readiness and workflow admission.

`BeginClassifiedRecovery` requires the exact active `EngineFailureSequence` and the one
`EngineRecoveryRequest` allowed by that disposition. Its successful result and replacement event
contain the exact `EngineRecoveryAction` needed by the subsystem worker. `CompleteRecovery` clears
the active record only after that complete token matches. `FailRecovery` routes the failed lifecycle
action back through the coordinator, assigns a new monotonic failure sequence, retains the attempt
count, and reclassifies the next allowed action. A stale action changes no lifecycle, recovery
revision, command sequence, or event stream.

`InspectRecoveryState` returns the same complete state without publication. The independent recovery
revision advances once for every successful report, begin, complete, or fail transition. Diagnostic
history remains bounded and ordered, and its raw internal evidence stays distinct from the
coordinator's reviewed user-safe projection.

## Playback command bridge

`PlaybackTransportCommand` is the stable typed vocabulary for inspection, play, pause, exact seek,
scrub begin, scrub replacement, scrub completion, loop changes, signed rate, direction, and exact
frame steps. It contains only exact media values and policy choices. Process-local clock instants
come from the playback executor, while audio samples, frame values, textures, and packet bytes stay
outside the command protocol.

`EngineCommandDispatcher::new_with_playback_bridge` keeps authoritative command and lifecycle
admission on `EngineControl` and returns a separate `PlaybackCommandExecutor` for the real
`PlaybackTransport` owner. The bridge is bounded to one command and uses nonblocking send and
receive operations. EngineControl admits each mutating command against the current playback work
requirements, reserves its event identity, and will not accept another state-changing command until
that completion is collected. Read-only engine inspection remains available while playback is
executing.

The executor applies the command only on `ExecutionDomain::Playback` and returns the resulting full
transport snapshot. A real transport method can fail after cancelling stale work or making another
externally visible discontinuity change, so an accepted command always produces a replacement
event. Failure events include bounded category, recoverability, summary, and context evidence
instead of claiming rollback that the transport owner cannot guarantee. Transaction identity,
originating command sequence, event sequence, lifecycle permit, and transport epoch remain
correlated across the domain boundary.

## Export command ownership

`EngineExportJobCommand` is the stable typed vocabulary for submit, poll, one-job and all-job
inspection, pause, resume, retry, one-job or all-job cancel, and remove. Submit validates at most
64 deterministic direct dependencies and refers to a previously prepared `JobId`; generic executor
closures and typed artifacts never enter command or event payloads. `EngineExportJobState` carries
one monotonic export state revision and complete deterministic snapshots for every retained job, including
status, attempt, progress, elapsed time, dependencies, failure evidence, result availability,
retry policy, and finality.

`EngineCommandDispatcher::attach_export_jobs` installs one type-erased controller over the
canonical `ExportJobQueue<R>` and returns an `EngineExportJobRuntime<R>`. The runtime handle only
prepares a typed executor, reads a completed typed result, and performs the queue's explicit
blocking-safe shutdown after dispatcher state is final. Every logical queue mutation still enters
through the dispatcher on EngineControl. A nonblocking ownership lock rejects accidental concurrent
access instead of waiting on the serialized control domain.

Submit, resume, and retry require fresh coherent export admission. The dispatcher supplies the
exact current `EngineWorkPermit` to the executor for each new attempt, so a recovery retry cannot
reuse its pre-degradation permit. Poll, inspect, pause, cancel, and removal remain available while
export is degraded, allowing automation to observe and settle retained work without admitting new
execution. Poll compares complete queue state before and after the canonical queue refresh, so
worker start, progress, completion, failure, dependency release, pause acknowledgement, and cancel
acknowledgement become ordered replacement events without a second job owner.

The prepared executor can call `render_and_export`, which validates the supplied export permit at
admission and immediately before complete artifact publication. Elementary packet streams and
typed artifacts remain behind the runtime handle rather than moving through the command boundary.
The all-job cancel command plus polling makes every retained job final before worker joining.
Joining then changes no logical job state and remains a blocking-safe finalization step after
logical lifecycle shutdown because EngineControl itself may not block.

## Public API and headless automation

The public scenario API adds the permanent method
`superi.slice.scenario.transaction.execute` and the permanent event name
`superi.slice.scenario.state.changed`. `ExecuteScenarioTransaction` carries the transaction ID,
expected project revision, and strict typed actions. `ScenarioTransactionResult` returns the same
identity, successful command sequence, committed action kinds, and complete state. Each committed
mutation emits `ScenarioStateChanged` with its own ordered event sequence plus matching command
sequence, identity, revision, and full state.

`ScenarioApi` now owns the scenario-only engine dispatcher. It converts public actions before
dispatch, preserves structured engine error category, recoverability, context, and last valid
state, and exposes ordered event draining. The legacy `ExecuteScenarioAction` command wraps a
single-action transaction and therefore cannot drift into a competing mutation implementation.

The canonical CLI now sends revision-fenced transactions through that public API. After each
mutation it drains exactly one state event and verifies transaction identity, command correlation,
revision, and complete state before continuing. Existing report state and stage behavior remain
unchanged, while the first production Rust consumer now proves the command and event seam rather
than merely calling the reference state owner.

## Research basis

The JSON-RPC 2.0 specification informed stable namespaced method identity, caller-owned request
identity, structured success or error outcomes, and transport-neutral request processing:
<https://www.jsonrpc.org/specification>.

JSON-RPC batches may be processed in any order and are not an atomic transaction primitive. Superi
therefore uses one typed transaction request with ordered actions and one expected revision rather
than treating a wire-level batch as an engine commit boundary. No network transport dependency was
added.

## Verification

Engine contracts were written first and failed because no dispatcher module or types existed. The
completed four-case dispatcher suite proves multi-action rollback, one-revision commit, one-unit undo, stale
revision rejection, ordered full-state events, rejection before mutation at the 64-event capacity,
off-domain rejection, normal and degraded playback, rendering, and export admission, recovery,
stale permit invalidation, retained failure context, restart, shutdown, and inspection.

After adjacent transport and render-export checkpoints landed, a new bridge integration contract
was written first and failed because the transport command vocabulary, bridge, executor, accepted
result, epoch correlation, and playback replacement event did not exist. It now drives the real
bounded worker, prefetch, audio discard, clock, and viewport transport harness across EngineControl
and Playback. The proof covers normal exact seek, rejection of an overtaking state command,
playback degradation denial, lifecycle recovery, and a failed exact seek that retains unchanged
state plus classified failure evidence in its ordered event.

After the logical export queue landed, a three-case export dispatcher contract was written first
and failed because the stable export command, state, controller, runtime handle, result, and event
surfaces did not exist. It now runs the real bounded export worker queue and proves prepared submit,
typed result retention, complete state and revision events, progress and completion polling,
inspection without events, pause acknowledgement, fresh resume, dependency release, cancellation,
degradation denial, observation during degradation, recovery retry with a distinct permit, removal,
all-job cancellation, and final-state-only blocking-safe shutdown. The original seven export queue
contracts and real render-export queue consumer remain green beside the dispatcher proof. The real
paired render-export consumer now
prepares its artifact executor through the typed runtime handle, submits and polls exclusively
through dispatcher commands, verifies replacement events, and reads the retained typed artifact
only after dispatcher-visible completion.

After the classified error coordinator landed, a two-case error dispatcher contract was written
first and failed because the bounded operation label, classified commands, composite state and
result, recovery event, and revision correlation did not exist. It now drives the real coordinator
and lifecycle through retryable and degraded failures. The proof covers exact request validation,
coherent unrelated-work admission, full diagnostic and active-state replacement, inspection without

stale-token rejection, and a fresh second recovery.

Public API contracts were also written before the transaction command, result, event, method names,
and facade methods existed. They now prove strict JSON round trips, permanent names, atomic failure,
revision fencing, event and result agreement, inspect behavior, undo behavior, and legacy command

Focused engine, API, and CLI contracts, every affected package test and documentation test, strict
all-target Clippy, codebase-map validation, and the repository final verifier form the completion

## Remaining constraints

The dispatcher is an in-process engine control plane, not a network server. Wire transport,
authentication, subscriptions, cancellation, retry and idempotency policy, version negotiation,
general editor commands, scripting runtime, UI, extension host, and closed-tier clients remain with
their separate owners.

The public JSON transaction currently projects the canonical scenario model only. The engine Rust
command boundary now routes the real playback transport and logical export queue, but wire methods
and subscription projections for those commands plus classified error diagnostics remain with the
unified public protocol and application host. Executor preparation is runtime resource binding, not
a serializable command, and typed results remain process-local. The dispatcher does not yet make production project
persistence, generic timeline and graph mutation, cache invalidation, render execution, export
muxing, or publication one cross-subsystem atomic commit. Lifecycle actions still require real
subsystem owners to perform their nonblocking work before acknowledging tokens, the logical export
queue is not crash persistent, render-export still returns elementary streams without persistence,
and the canonical CLI remains contract conformance rather than a playable editor export.
