# P3.W04.C015 precise audio-video editing

## Outcome

The production timeline now links video and audio clips, synchronizes audio by exact source time,
detaches audio without moving it, replaces audio through the existing source-monitor path, and
publishes complete per-source-channel routing. The visible Audio and video panel keeps exact sample
timing, semantic channel order, relationship intent, destination intent, pending state, and command
results discoverable without creating another authored mutation owner.

## Native editorial ownership

`superi-timeline` owns three new atomic edit operations for audio-video link, exact synchronization,
and detach. Every operation resolves one clip on a video track and one clip on an audio track,
checks both locks and relationship admission, and publishes through the existing revision-fenced
edit batch. Link and detach change only canonical link intent. Synchronization preserves the audio
record range and duration, converts the video source anchor plus the exact record offset into the
audio source clock, translates the complete audio time map, and rejects any inexact conversion.

Ordinary replace remains the only replacement operation. Its structural reconciliation now moves
selection, link, group, and multicam intent from the removed clip identity to the caller-owned
replacement identity, while the existing engine clip-mix reconciliation moves authored clip gain,
pan, mute, and send state in the same compound transaction.

One new track mutation replaces complete audio routing. It preserves the track sample rate and
ordered source layout, requires one channel or mute decision for every source channel, validates
the destination layout, and permits only main output or a different existing audio track. Missing,
duplicate, reordered, self-referential, non-audio, and semantic no-op requests fail without
publishing a partial route or project revision.

## Public and desktop integration

`superi-api` exposes strict `link_audio_video`, `synchronize_audio_video`, `detach_audio`, and
`set_audio_routing` DTO variants through the unchanged `superi.project.command.execute` method.
The project editor schema is 1.7.0 and the complete public catalog is 1.9.0. Evidence retains the
exact operation kind, and the generated TypeScript artifact is derived from the Rust registry.

The desktop canonical projection retains audio sample rate, ordered source channels, destination,
destination channels, and every route instead of discarding them. Pure planners resolve the exact
video and audio selection by immutable track identity, compile public edit actions, and reject an
incomplete routing draft. The React panel exposes Link audio and video, Sync by source time, Detach
audio, Replace audio, track and destination choices, and one mapping or mute choice per source
channel. Link, synchronization, detach, and routing use the application action executor, while
replace reuses the exact source-monitor command executor. Transaction identity, revision fencing,
native validation, history, durable publication, correlated events, and complete snapshot refresh
stay with existing owners.

## Failure and compatibility behavior

Existing project and timeline serialization remain compatible because clip links, exact time maps,
audio routing, and replacement identities already belong to canonical state. The additive public
unions receive minor version releases, while permanent method, event, resource, timeline document,
project document, and transport identities remain unchanged.

The UI never infers a route, rounds a clock, patches canonical relationship state, or treats display
track indices as identity. Unsupported selection, a locked track, stale revision, invalid
relationship state, incomplete channel coverage, invalid destination, unsafe exact conversion, or
lower publication failure remains visible and leaves the last valid project snapshot untouched.

## Remaining constraints

This checkpoint authors timeline relationship, source-time, replacement, and routing intent. It does
not claim decoded audio binding, live device rerouting, variable-rate audio playback, plugin signal
processing, or sample rendering. Those runtime consumers remain with the engine and audio
subsystems and must preserve the exact canonical intent delivered here.
