# P3.W04.C016 editorial viewer and audio feedback

## Outcome

The editing workspace now publishes immediate trim, slip, slide, multicam, and audio consequences into the source viewer, program viewer, and one ordered routing and continuity meter rack. Every feedback value is derived from the active exact edit plan or the canonical replacement snapshot, remains presentation-only, and keeps unavailable runtime evidence explicit.

## Architecture and integration

`timeline-editorial-feedback.ts` is one pure immutable projection boundary over the active canonical canvas model, strict clip detail, exact public operation plan, and canonical editor audio state. Slip feedback shows the proposed source start while retaining the record range. Slide feedback retains the source range while showing the proposed record start. Trim feedback retains pre-edit source evidence while showing the exact proposed record boundary. Other tools retain their exact plan label and canonical source and record ranges without duplicating lower editing algorithms.

`TimelineWorkspace` remains the transient gesture owner and the existing project command path remains the only authored mutation owner. The workspace publishes one replacement feedback value for idle, preview, applying, applied, failed, and unavailable states. `ApplicationProvider` owns that cross-sibling presentation value beside the existing source-monitor snapshot, and `EditingWorkspacePanel` passes it into the two viewer shells and the meter consumer without creating another context, reducer, transport, or project state.

`NativeViewport` continues to publish only role, geometry, scale, visibility, and status through the native placement command. Editorial feedback renders after the native child placement host, so it remains visible without sending clip, multicam, audio, meter, pixel, or frame material through Tauri IPC. Source and program strips show exact coordinates and consequences, while multicam feedback retains sync method, complete angle identity and enabled state, exact switch ranges, and full audio policy.

## Audio truth and continuity

The audio feedback rack preserves each canonical track sample rate, ordered source and destination channel identities, explicit source-to-channel or muted routes, destination intent, clip count, and enabled, mute, and solo behavior. Audited seams keep seamless, gap, overlap, continuous-source, discontinuous-source, and different-source meanings plus exact sample counts and coordinates. Unsupported continuity stays unsupported.

The public editor snapshot does not expose live RMS, sample peak, true peak, or loudness observations. The meter contract therefore publishes `signalStatus: "unobserved"`, labels live level as unobserved, and uses bars only to distinguish route and audibility states. It never turns absent telemetry into a fake numerical level or claims that an unobserved route is silent.

## Multicam validation and failure behavior

Clip presentation now retains complete canonical angle and switch detail in addition to the existing sync, switch-count, and audio-policy summary. Duplicate angle identities, switches to missing angles, fixed-audio policies aimed at missing angles, multicam clips without source state, unsupported policies, malformed ranges, and malformed canonical state keep the supplemental projection unavailable instead of publishing contradictory viewer feedback.

A missing active clip produces explicit source-unavailable and program-playhead feedback while preserving any canonical audio track evidence. A missing audio track presentation produces unavailable audibility and route states. Disabled, muted, solo-suppressed, routed, muted-route, and unrouted states remain distinct. Viewer feedback cannot alter native presentation, and every authored edit still requires a successful project command followed by canonical refresh.

## Verification

Pure contracts cover trim, slip, slide, multicam angle and switch fidelity, fixed audio policy, mixed 48 kHz and 96 kHz sample clocks, ordered channel meaning, routing, solo suppression, gap and discontinuous-source seams, deep immutability, explicit unobserved signal telemetry, and missing clip behavior. Clip projection contracts reject missing multicam angle references. Application ownership and native IPC contracts prove real viewer and meter wiring without a second authored owner or an expanded placement payload.

Strict TypeScript checking, the complete 66-test production frontend suite, and the Vite production build exercise the real editing consumer. Codebase-map validation and the repository checkpoint verifier cover the final delivered diff.

## Compatibility boundary

No public API schema, timeline operation, project history behavior, native viewport command, engine owner, or project format changes in this checkpoint. Existing multicam summary fields remain available, new viewer props are optional, canonical track and route ordering is unchanged, and feedback remains a transient deeply frozen projection over existing authoritative state.
