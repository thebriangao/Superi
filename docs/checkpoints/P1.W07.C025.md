# P1.W07.C025

## Outcome

Added bounded timing and process resident-memory instrumentation to every stage of the canonical
first editorial slice. Each of the eight stage records now contains monotonic elapsed microseconds
and resident-set bytes sampled before and after the stage. Report schema 1.1.0 also declares the
measurement contract and the maximum resident value observed at those boundaries.

## Architecture and integration

`superi-cli` owns instrumentation because it already owns stage sequencing and strict report
serialization. A private sampler resolves the current process once, refreshes only that PID with
memory enabled and task enumeration disabled, and takes exactly two samples per stage. The sampler
does not enter `superi-api` or `superi-engine`, alter scenario state, scan unrelated processes,
spawn a background thread, allocate an unbounded trace, or introduce a network path.

The existing top-level `duration_us` field remains stable. Each stage gains a `memory` object with
`resident_bytes_before` and `resident_bytes_after`. The report gains an `instrumentation` object
whose fixed metadata names the monotonic clock, microsecond and byte units, process resident-set
metric, stage-boundary sampling, and instrumented stage count. Its
`observed_resident_bytes_max` value is the exact maximum of all 16 serialized boundary samples.

## Dependency decision

Rust's standard library provides the monotonic clock but no portable process resident-memory API.
The CLI therefore uses `sysinfo` 0.36.1 with default features disabled and only its `system`
feature enabled. That release is MIT licensed, declares Rust 1.75 support, and remains compatible
with the workspace's Rust 1.80 contract.

The runner treats an unavailable process identity or zero resident-memory sample as an explicit
`unavailable` stage failure. It never substitutes a fabricated zero or silently drops a memory
field from an otherwise successful report.

## Implementation process

The synchronized base was `218e110`. The report-contract test was changed first and failed on the
real CLI because the production report still emitted schema 1.0.0 and had no memory records. The
implementation then added one reusable sampler, wrapped all eight stage boundaries, preserved the
existing duration field through flattened serialization, and generated the instrumentation summary
only after every stage completed.

The canonical fixture resolver and state engine remain the real consumer path. Instrumentation
surrounds fixture validation, scenario API calls, deterministic state transitions, contract
artifact publication, and final undo and redo verification without replacing any of them with a
mock or benchmark-only route.

## Verification

The focused scenario contract runs the built `superi-cli` binary twice. It proves schema 1.1.0,
the exact eight-stage order, a serialized duration for each stage, two nonzero resident samples per
stage, fixed measurement metadata, stage count eight, and a summary maximum equal to the largest
serialized sample. It then removes only dynamic timings, memory values, the observed maximum, and
output paths before requiring the two reports and contract-stub artifacts to match exactly.

Fresh manual runs also inspect the complete instrumentation record and compare independently
normalized reports. Broader CLI, workspace, fixture, dependency, license, formatting, static, and
map validation evidence is recorded in the ignored execution log.

## Delivery

This record is committed with the sampler, stage integration, report contract, dependency lock,
tests, vertical-slice documentation, and current codebase maps. The delivery commit containing this
record is pushed to `origin/main`; its SHA is recorded in the matching Google Docs completion
description and ignored execution log after it exists.

## Remaining constraints

Boundary sampling records process resident memory immediately before and after each stage. The
largest observed boundary value is not a continuously sampled intra-stage peak, allocation trace,
hardware comparison, constrained-device threshold, or long-session soak result. Later performance
and platform checkpoints may add those proofs without changing the eight-stage editorial contract
or silently reinterpreting these fields.
