# P2.W07.C025: durable project command recording

Every successful `superi.project.command.execute` request is recorded in the project that it
affected. The record is part of durable project persistence but remains outside authored project
state, authored revision, semantic hashing, and undo history. Apply, undo, redo, inspect, and
authored no-op requests therefore share one stable typed command surface without turning debugging
metadata into editable project content.

## Atomic execution

The engine prepares request evidence before dispatch and executes the command against a candidate
history state. A successful command and its command-log append publish together. Conversion errors,
permission denial, stale revision fences, command failures, event backpressure, sequence exhaustion,
and invalid record evidence publish neither side.

Every successful generic command produces exactly one record and one
`superi.project.state.changed` event. The API result, event, and durable record correlate through
the engine command sequence, caller transaction identity, project command kind, and project-local
command-log sequence. Inspect and semantic no-op commands keep `authored_state_changed` false and do
not advance the authored revision, but they still advance the command-log sequence and publish the
replacement event needed by observers.

## Durable record contract

Each record contains:

- one positive monotonic project-local sequence;
- the correlated positive engine command sequence;
- the exact caller transaction identity and permanent public method name;
- the request schema version and apply, undo, redo, or inspect command kind;
- the caller's expected project revision;
- the exact compact request byte length and lowercase SHA-256 digest;
- retained or digest-only payload disposition;
- before and after authored project revisions;
- before and after semantic project hashes; and
- whether authored state changed.

Exact compact typed request bytes are retained when they are at most 1,048,576 bytes. Larger valid
requests remain executable and retain length and digest evidence with `digest_only` disposition.
The log retains at most 4,096 records and 67,108,864 retained bytes, evicting the oldest complete
records first while preserving monotonic sequence identity and explicit cursor gaps.

## Persistence and compatibility

Project schema 5 and semantic format `1.4.0` add strict `command_log_metadata` and
`command_log_records` tables. A schema 4 project migrates to schema 5 with an empty command log and
unchanged authored state. Current-schema save, save-as, copy, backup, autosave, reopen, integrity
validation, and migration use the same typed persistence owner.

The active project's log lineage survives recovery restore. Recovery candidate authored state may
replace active authored state through the existing checked restore transaction, but candidate log
records are never imported and the active sequence never rewinds.

## Public inspection

`superi.project.command_log.get` has schema `1.0.0` and accepts:

```json
{
  "after_sequence": 0,
  "requested_limit": 256,
  "detail": "metadata"
}
```

The requested limit is from 1 through 256. A valid cursor returns a bounded `records` result with
the requested cursor, through sequence, oldest available sequence, latest sequence, and ordered
records. A cursor whose required prefix was evicted returns `resync_required` instead of silently
returning a partial continuation. A cursor newer than the selected project is invalid.

`metadata` never returns retained request bodies. `replayable` decodes each retained request through
the current strict `ExecuteProjectCommand` type and reauthorizes its original filesystem and plugin
requirements before returning any result. One denied record fails the whole query, so replay detail
cannot partially disclose an unauthorized batch. Digest-only records remain inspectable but have no
replay request.

The public catalog is schema `1.4.0`, the generic project editor command, result, history, and event
contract is schema `1.1.0`, and the command-log query and replacement resource are schema `1.0.0`.
The generated TypeScript artifact exposes the request, result union, record types, event sequence,
method map entry, and `superi.project.command_log` resource map entry.

## CLI, automation, and scripts

The headless CLI reads the log through the public API rather than reading SQLite directly:

```text
superi-cli project command-log --project <PROJECT> --after-sequence <U64> --limit <U32> --detail <metadata|replayable> [--permissions <JSON>]
```

The same query is accepted by local JSON-RPC automation with method
`superi.project.command_log.get`. It is also a supported `superi-json` step, so a script can execute
a typed project command and inspect its resulting durable record through the same
`ProjectEditorApi`. Local hosts persist every successful generic command, including inspect and
no-op commands whose authored revision does not change.

The stable mutation vocabulary remains `ExecuteProjectCommand` and its closed `ProjectCommand` and
`ProjectAction` unions. The log is evidence for debugging and reproducibility, not a second command
bus, state owner, parser, or persistence store.
