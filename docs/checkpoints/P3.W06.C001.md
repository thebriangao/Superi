# P3.W06.C001

## Outcome

Implemented persistent native window creation, restoration, multi-window behavior, fullscreen,
monitor movement, and reversible placement recovery while preserving one shared engine and project
authority.

## Architecture and integration

`DesktopWindowState` owns one strict schema-versioned session with exactly one primary window,
bounded auxiliary workspace windows, stable labels, workspace routes, normal placements, fullscreen
state, monitor identity, previous placement, last focus, and a bounded recently closed stack. It
loads before application setup, reconciles missing or changed monitor topology into visible bounds,
preserves malformed state as a rejected artifact, publishes one explicit recovery notice, and
coalesces atomic background persistence without blocking the Tauri event loop.

The shell exposes typed snapshot, create, focus, fullscreen, monitor move, placement undo, workspace
update, close, and reopen commands plus one ordered replacement event. Auxiliary webviews reuse the
same application provider, project lifecycle, and engine process. Native transport generations and
pending requests are scoped by invoking webview label, while authored replacement events retain one
global sequence and are delivered to every connected client with that client's generation.

`WindowSessionPanel` is the real System panel consumer. It exposes exact active, focused,
fullscreen, monitor, placement, persistence, recovery, and recently closed state with reversible
controls. Restored workspace routes hydrate before normal route persistence begins. The primary
`main` window remains the sole owner of role-addressed native GPU viewer surfaces, so auxiliary
windows expose the same engine and workspace state without moving or duplicating native viewport
ownership.

## Implementation process

Focused Rust proof first failed because the session model, persistence owner, and placement
operations did not exist. Focused frontend proof first failed because the typed bridge and System
panel consumer did not exist. The implementation added the native owner and command registration,
made transport connection state webview-local, added the strict React bridge and panel, hydrated
the active workspace route, isolated native viewport ownership, and then widened proof to the full
desktop Rust and frontend suites.

## Remaining constraints

Window sessions retain application workspace routes and placement intent, not authoritative project
or editor documents. Monitor identities are local routing hashes derived from current Tauri monitor
facts and are reconciled when topology changes. Auxiliary windows intentionally do not own native
GPU viewer surfaces because the current presenter retains one role-addressed surface set attached to
the primary webview; they remain fully connected control and state consumers.
