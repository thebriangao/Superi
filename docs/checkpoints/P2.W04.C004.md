# P2.W04.C004

## Outcome

Implemented complete clip gain, exact linear fades, equal-power stereo pan, mute, project-wide
solo, per-channel phase inversion, and explicit semantic channel mapping. Clip mix intent is
revisioned and follows retained, fragmented, replaced, and removed clip identities without
synthesizing state for clips that never owned explicit controls.

## Architecture and integration

`superi-audio::mixing` owns `ClipMixControls`, the source-to-destination channel matrix,
transactional `ClipMixState`, immutable `ClipMixSnapshot`, and the prepared `ClipMixProcessor`.
Editable state validates finite bounded coefficients and controls outside the callback. Snapshot
preparation resolves the project-wide solo decision, validates exact clip sample bounds and fade
duration, and precomputes routing and phase coefficients.

The processor implements the existing `AudioProcessor` contract. It validates the prepared sample
clock, ordered channel layouts, and half-open clip interval before touching output, then applies
channel mapping, the W3C stereo panner equations, bounded linear gain, endpoint-inclusive sample
fades, phase, mute, and solo without allocating or locking on the successful callback path.

`superi-engine::audio_mix` is the upward integration owner because audio cannot depend on timeline.
It applies a real timeline edit batch to cloned project state, translates typed edit outcomes into
audio fragment inheritance, replacement transfer, and removal, validates the mix revision, and
publishes both subsystem states only after both succeed. Stable-identity timing changes require no
mix mutation, so source, record, timing, identity, grouping, synchronization, and user intent keep
their established owners.

## Implementation process

Public audio contracts were added first and failed on the missing mixing API. The implementation
then added complete control validation, transactional identity mutations, snapshot preparation,
and DSP through the existing prepared graph. Focused proof exposed a trigonometric residue at hard
pan endpoints, so `-1` and `1` now use exact branch results while interior values retain the
equal-power equations.

The engine consumer contract was also introduced red against a missing reconciliation seam. The
first integration supported typed timeline outcomes, then the contract was strengthened to require
one atomic project-and-mix publication so revision conflicts cannot commit only one side.

Primary-source review used W3C Web Audio 1.1 for per-sample gain, linear ramps, equal-power stereo
pan, and explicit channel composition, plus Apple Audio Unit render guidance for monotonically
consecutive sample timestamps, prepared buffer topology, and allocation before real-time rendering.

## Verification

`clip_mixing_contract` processes real prepared graph blocks at 48 kHz. It proves swapped semantic
channel routing, phase inversion, gain, exact three-sample fade endpoints across two callbacks,
hard pan, mute, project-wide solo, atomic set, inherit, transfer, and removal, stale revision and
partial-batch rejection, invalid numeric and layout state, fade-duration bounds, and out-of-clip
processing rejection.

`audio_editorial_mix_contract` drives a real `superi-timeline` razor operation through the engine
transaction. It proves exact source and record subdivision, retained and new identities, group,
link, and sync-lock inheritance, complete audio intent inheritance, and unchanged project and mix
state when the audio revision conflicts. Affected package tests, warnings-denied Clippy,
formatting, dependency direction, map validation, public documentation, and the repository verifier
are recorded in the ignored execution log.

## Delivery

This record is delivered with the audio implementation, engine reconciliation seam, public

## Remaining constraints

Gain is linear rather than decibel-addressed, fades are linear only, and nonzero pan targets the
canonical stereo output. This checkpoint does not implement multi-input buses, automation curves,
decoder or device playback, metering, resampling, plugin hosting, or final mix export. The engine
transaction is a Rust orchestration surface and is not yet exposed through the public API or a
playback controller.
