# P3.W06.C016

## Outcome

Implemented project-aware desktop window titles, semantic dirty-state presentation, and safe project
replacement, operating-system open, close, window-close, and quit behavior. A modified project is identified from authored
project meaning relative to its latest explicit save point, so undoing back to saved content clears
the indicator even when session redo history remains.

## Architecture and integration

The existing `superi-project` semantic hash remains the content identity authority. The API editor
snapshot now exposes its root timeline identity to native consumers, and the desktop project
lifecycle observes one coherent document identity plus versioned SHA-256 semantic hash after every
create, open, save, save-as, recovery, settings, import, and editor transaction. The lifecycle keeps
the current and saved semantic hashes private, persists only the saved baseline beside the active
path in schema 2 session continuity, and derives the public `dirty` boolean without treating revision
or undo depth as modified content. Schema 1 sessions migrate by accepting the current reopened
document as their clean save point.

The native desktop shell receives only the derived dirty bit with the existing document and history
presentation. It adds a familiar title marker to modified documents, disables Save for clean or busy
state, rejects dirty state without an active document, and keeps dirty state out of private workspace
persistence. The React shell requires exact project, root timeline, and revision agreement for
history actions, shows document state in System, and applies one shared close decision matrix to
project replacement, explicit close, window close, and quit. Modified content is saved before the
operation continues, save failure cancels the transition, busy or synchronizing state fails closed,
and clean session history still receives an explicit history-loss confirmation without a redundant
save.

Startup arguments and macOS resource-open events no longer execute a native project replacement.
They enter a bounded process-local request queue with monotonic identity and explicit acknowledgement.
The primary React shell subscribes before reading the retained queue, deduplicates event and snapshot
delivery, waits for coherent project and editor state, then routes each request through the same safe
replacement owner. A cancelled prompt, busy state, failed save, or failed open preserves the prior
active project, and auxiliary webviews cannot race the primary document owner.
