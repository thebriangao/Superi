# P3.W03.C012 batch media operations

## Outcome

The desktop media library now commits ordered batch rename, organization, optimized transcode, proxy, relink, and metadata operations as one revision-fenced replacement transaction. Editors can select many visible sources, run the supported operations from the real media browser, inspect the resulting state, and immediately search updated names and metadata.

## Architecture and integration

`DesktopProjectState` remains the native persistence and mutation owner. `MediaBatchUpdate` applies a bounded nonempty operation list to a cloned `MediaLibrarySnapshot`, validates the complete candidate, advances the library revision once, then publishes one replacement snapshot and durable sidecar update. Any invalid operation rejects the complete list with its zero-based operation index and leaves the prior authoritative state intact.

Rename changes only the durable presentation name, which is retained when the same stable import identity is projected again. Organization reuses the delivered bin-move rules. Relink reuses the C009 fingerprint guard, so paths may change without changing source identity or freshness. Metadata reuses the C005 key, value, capacity, and removal validation.

Proxy and transcode operations reuse the C008 `DerivedMediaAttachment` lifecycle with fixed proxy or optimized purpose, explicit quality, status, source fingerprint, source revision, byte length, and optional representation choice. Generating, failed, stale, absent, or mismatched derived media resolves deterministically to the authoritative original source. Actual codec execution and artifact byte production remain owned below the desktop media-management layer; this checkpoint batches transparent lifecycle evidence and never reports a generating artifact as ready.

## Consumer behavior

The TypeScript bridge exposes the exact operation union, revision-fenced update, result evidence, and Tauri invocation. The production React media browser offers multiple selection, deterministic numbered rename, active-bin organization, generating quarter-quality proxies, generating full-quality optimized transcodes, root-based path relink, and metadata upsert or removal. Each successful action reports the committed operation count, affected stable identities, and replacement library revision.

Local search now includes source facts and user metadata in addition to names, paths, built-in metadata, annotations, and offline state. Native replacement snapshots serialize refreshed usage, identity, smart-collection membership, thumbnails, offline availability, and representation resolution so the real consumer receives the state it presents.

## Verification

The frozen `media_batch_operations_contract` exercises a real two-source project, a nine-operation commit, one revision advance, stable identity and freshness, generating proxy and optimized fallback, complete rollback after a later invalid operation, and sidecar restoration. Adjacent C004 through C009 contracts, Rust formatting, targeted lint with the known transport error-size exception, frontend typecheck, frontend tests, the production build, the refreshed codebase-map validator, and the repository checkpoint verifier pass. The unrestricted Rust lint and full unit-test suite still report pre-existing transport diagnostics outside this checkpoint: twelve large-error warnings and one schema-name fixture failure.

## Remaining constraints

Batch relink accepts a caller-selected root and preserves each source basename while relying on the delivered caller-supplied fingerprint contract. Proxy and optimized controls create explicit generating lifecycle entries; a future engine-side producer must attach ready artifact evidence through the same freshness-fenced contract before those representations become active. Batch replacement and conform are intentionally excluded because C012 names relink while C009 already owns deliberate source replacement and rate reinterpretation as distinct operations.
