# P1.W05.C004

## Outcome

Superi now loads bounded, versioned color-management files and pins each project working-space selection to the exact validated configuration semantics.

## Architecture and integration

The `superi-color` crate owns the strict JSON boundary, immutable named scene-linear working spaces, aliases, roles, and deterministic SHA-256 fingerprint. `ProjectColorSettings` serializes the canonical selection and refuses to resolve against a config whose stable identity or semantic fingerprint has drifted. The canonical color fixture now selects ACEScg through this configuration boundary before exercising the authoritative input and output transforms; display rules, looks, ICC evaluation, and graph execution remain with their existing or later owners.

## Implementation process

The implementation followed the existing `WorkingSpace`, `ColorSpace`, deterministic collection, structured error, and bounded-input conventions from verified base `07278a408c4e55aad2788042520bb5b156e35cae`. Contract tests were added first around the public configuration and project-selection behavior, then the placeholder module was replaced, dependencies were scoped to `superi-color`, file reads were hard-bounded against size races, and the result was reconciled with the authoritative C003, C005, C010, and fixture APIs.

## Remaining constraints

The file format intentionally supports only the named scene-linear spaces, aliases, and roles required by this checkpoint. Unknown schema fields, unsupported versions, nonlinear working spaces, ambiguous names, limited-range RGB, oversized files, and config drift fail closed rather than being approximated.
