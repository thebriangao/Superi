# P3.W05.C010

## Outcome

Implemented selectable external display output for source, program, composite, and color viewers wherever the desktop monitor and GPU surface path permits it.

## Architecture and integration

The native viewport owner projects Tauri's current monitor catalog into deterministic connection-local routing targets, excludes the display containing the editor window, rejects target sharing between viewer roles, and reports exact target geometry, scale, selected and presented analysis view, surface generation, frame sequence, display intent, and failure state. Each role owns one hidden borderless external window and wgpu surface beside its existing inline child, while the sole GPU submission thread presents both destinations from the same managed canonical RGBA16F render result through the same `GpuDisplayPresenter`; external failures remain isolated from inline presentation and no media enters Tauri IPC.

`NativeViewport` adds one accessible transient display selector without changing navigation, overlay visibility, comparison captures, analysis selection, temporal context, or the existing exact timecode, source, playback, visual, audio, comparison, and editorial status projections. The editor shell continues to communicate those states while the external window provides clean native image output, and stale or disconnected targets fail closed to an explicit unavailable state.

## Implementation process

Focused frontend and native contracts first failed on the missing shell state owner, monitor catalog, destination enum, and strict display-selection field. The implementation then added frozen target selection and status formatting, pure monitor projection, per-role external native windows and surfaces, destination-aware GPU commands, real viewer controls, target ownership conflict handling, hotplug reconciliation, and external presentation failure isolation.

## Remaining constraints
