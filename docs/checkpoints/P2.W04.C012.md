# P2.W04.C012 VST3 effect hosting

## Outcome

Superi audio now hosts one explicit VST3 audio-effect class inside a dedicated plugin worker on
macOS, Windows, and Linux. The prepared graph processor preserves canonical channel meaning, exact
sample time, sample-offset automation, bounded monitoring, real-time or offline process mode, and
audible source-to-master delivery.

## Architecture and integration

`superi-audio::hosting::vst3` owns the safe configuration, strict class identity, immutable plugin
metadata, bounded control-to-audio automation writer, prepared graph effect, output-parameter
readings, telemetry, and explicit worker session. Its private `native` module retains the platform
module, VST3 factory, component, processor, optional controller, host callbacks, fixed parameter
queues, planar f32 bridge, and reverse lifecycle. The session refuses to unload while a prepared
graph node still holds a process lease.

Initialization tracks each connection direction and each main-bus activation independently, so a
failure after one successful stage unwinds only completed ownership in reverse. Explicit shutdown
is fail-fast and retryable. If any stop, deactivation, disconnect, handler-removal, or termination
call fails, the session retains the remaining COM owners and mapped module until a later successful
retry or dedicated worker-process exit.

The host context also exposes the supported component-handler interface plus bounded in-memory
`IMessage` and `IAttributeList` objects for control-side component and controller communication.
Those objects reject audio-domain access before allocation or locking. Plugin-declared optional
process-context requirements are rejected because this subset supplies exact sample rate and project
sample time but does not guess tempo, musical position, transport, frame rate, or other context.

The supported effect subset is intentionally exact: one main audio input, one main audio output,
no event buses, f32 samples, and canonical mono, stereo, quad, 5.1, or 7.1 music arrangements.
Unsupported instruments, auxiliary buses, sidechains, topology, sample precision, or arrangements
fail before publication. Reported latency and tail remain visible metadata; this checkpoint does
not apply delay compensation.

Preparation caps planar sample storage and each automation or monitoring handoff at 1,048,576
elements, controller parameters at 4,096, and combined fixed parameter-point cells at 1,048,576.
Per-block automation cannot exceed its total queue. Oversized plugin metadata or caller bounds fail
before graph publication instead of relying on allocator failure.

## Timing, automation, and monitoring

Every graph block retains its integral `SampleTime`, sample rate, frame count, ordered
`ChannelLayout`, process mode, and exact VST3 `projectTimeSamples`. Preallocated interleaved-to-planar
and planar-to-interleaved copies preserve the semantic channel order. Automation submission
validates a complete nondecreasing slice before admission, retains one future point, and maps each
in-block point to its exact VST3 sample offset. Only parameters declared automatable and writable
are admitted; read-only parameters remain visible as metadata and output monitoring. Plugin-originated
parameter points map back to exact absolute sample coordinates through a bounded output ring whose
overflow remains observable. Fixed output queues are visited once in queue and point order, so
monitoring extraction remains linear in the plugin's bounded output rather than rescanning queues.

The audio callback performs no module lookup, controller or lifecycle call, allocation, free,
lock, wait, I/O, or formatted logging on its successful path. Nonfinite input is rejected before
the native call; native failure or nonfinite output clears the block and publishes a classified
error instead of partial final-mix audio.

## Platform boundary

macOS uses a retained `CFBundle` with `bundleEntry`, `bundleExit`, and `GetPluginFactory`. Windows
uses a retained library with optional `InitDll` and `ExitDll` plus the required factory. Linux uses
a retained `dlopen` owner, passes its actual handle to `ModuleEntry`, and pairs it with
`ModuleExit`. Exact low-level bindings are pinned to `vst3` 0.3.0 and
`com-scrape-types` 0.1.1, both licensed `MIT OR Apache-2.0`; the platform loaders reuse pinned
`libloading` 0.8.9 and `objc2-core-foundation` 0.3.2.

Primary behavior and lifecycle references are the Steinberg VST3 SDK 3.8.0 module architecture,
audio-processor call sequence, bus-arrangement sequence, speaker-arrangement definitions,
parameter-automation contract, and threading guidance. The raw boundary and its ownership rules
are inventoried in `docs/unsafe-ffi.md`.

## Verification

The public contract compiles a real VST3 dynamic-library fixture with the pinned bindings into a
temporary standards-shaped bundle. The parent never loads native code. Isolated child processes
prove mono through 7.1 channel identity, source to hosted effect to submix to master routing,
two exact in-block gain changes, one block-end point retained for the next block, real-time and
offline modes, process-context values, output monitoring and bounded overflow, latency and tail,
host message and attribute behavior, audio-domain host-object rejection, read-only automation
rejection, zero fixture-side callback allocations, explicit shutdown conflict, reverse teardown,
module exit evidence, fail-closed rejection of unsupported tempo context, exact unwind after partial
bus activation, and mapped-code retention when processor stop fails.

Focused VST3 proof, the complete audio package, format checking, warnings-denied Clippy, Rust 1.80,
dependency and license policy, supported-target compile checks, map validation, and the full
checkpoint verifier are delivery gates.
