# P1.W07.C002

## Outcome

Extended every existing continuous integration lane with Rust formatting, locked workspace tests,
strict Clippy, and locked documentation tests. The existing locked workspace builds remain part of
the same suite so contributors receive one consistent quality boundary before changes spread.

## Architecture and integration

The checks extend `.github/workflows/ci.yml`, the repository-owned CI boundary introduced by
P1.W07.C001. Both the cross-platform matrix and the scheduled Ubuntu 22.04 job install the declared
rustfmt and Clippy components, run from the `open/` workspace, and execute the same build,
formatting, test, lint, and documentation-test categories.

Linux lanes use `.github/scripts/provision-linux-libva.sh` to install exact source-build tools,
download the official libva 2.22.0 archive, verify its pinned SHA-256 before extraction, build a
private prefix, install the GBM development link target, and publish the header, pkg-config, native
linker, and runtime library paths required by the locked media graph. The same helper installs `nasm`
for the approved libvpx runtime's optimized x86 path.
Intel macOS lanes install `nasm` with Homebrew. Linux and macOS then build checksum-pinned libvpx
1.16.0 and expose the exact shared library to strict capability and codec contracts. Windows uses
the pinned vcpkg registry revision shipped with its hosted image to build libvpx 1.16.0 with VP9
high-bit-depth support as a dynamic MinGW runtime, then publishes the single discovered DLL path.

Hosted macOS runners skip only the three native VideoToolbox and AudioConverter lifecycle tests
that require the physical macOS hardware lane documented in `docs/platform-testing.md`. Linux and
Windows continue to run the exact locked workspace test command, and no test source, assertion,
feature matrix, or frontend coverage was weakened or added.

## Verification

The workflow was checked for valid YAML, exact command coverage in both jobs, full Linux and
Windows workspace tests, the three precisely scoped hosted macOS native skips, and the physical-lane
rationale. The focused workflow contract, repository map validator, diff checks, prohibited-dash
scan, and workspace formatting passed on the synchronized low-risk verification base. Results from
the formerly shared Cargo target were invalidated and are not used as delivery evidence; the
delivered GitHub-hosted workflow run is the authoritative execution proof for the installed Rust
commands.

The first delivered hosted run exposed absent `libva.pc` metadata on Ubuntu 24.04 and Ubuntu 26.04
before tests began and absent approved libvpx 1.16 runtime discovery on both macOS lanes. That red
evidence drove Linux-only `libva-dev` provisioning plus checksum-pinned libvpx runtime builds and
their focused workflow contracts; completion requires the follow-up hosted matrix to pass.

The next hosted run verified the source checksum and completed the arm64 macOS libvpx build, then
showed that x86_64 Linux and Intel macOS require `nasm` before configuring libvpx. That red evidence
drove explicit assembler provisioning rather than disabling optimized runtime code.

C009 hosted run `29308007012` then proved the assembler and libvpx corrections but exposed that
Ubuntu 24.04's distribution libva API 1.20 cannot satisfy the unchanged H.266 API 1.22 build
boundary. The shared checksum-pinned libva 2.22 source provisioner is the corrective path for both
workflows; no platform crate, feature, or API requirement was removed.

After GitHub Actions billing authority was restored, descendant run `29379237046` reached the
shared provisioner and exposed that Ubuntu 24.04's Meson 1.3.2 rejects the newer `--directory`
spelling for compile and install subcommands. The helper now uses the portable `-C` spelling, and
the focused workflow contract requires both exact commands while rejecting the incompatible form.
The follow-up run compiled and installed libva successfully, then exposed that the project release
version 2.22.0 provides VA API version 1.22.0 through pkg-config. The final helper assertion now
checks the unchanged required API version instead of comparing the unrelated release number.

Descendant run `29379849223` then passed the complete pinned source helper on both Ubuntu lanes and
reached the locked workspace tests. Those tests supplied focused red evidence that the test linker
could not discover the private `libva` and `libva-drm` libraries or the system `libgbm` target. The
helper now installs `libgbm-dev` and publishes its private prefix through `LIBRARY_PATH` for native
linking while retaining `LD_LIBRARY_PATH` for runtime discovery.

The next descendant run passed every Ubuntu step with the corrected native-link environment and
reached the strict Windows capability test after C003 corrected its production ABI width. Windows
then supplied red evidence that no approved libvpx runtime was installed. The Windows lane now
builds the exact 1.16.0 port from a pinned registry baseline with binary-cache reuse disabled, keeps
VP9 high-bit-depth support, and exposes the resulting DLL instead of weakening the capability test.
