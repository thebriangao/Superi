# P3.W03.C010 generated media previews

## Outcome

The desktop media library now generates bounded, replaceable thumbnails, ordered filmstrips, channel-separated waveforms, and inspector previews on demand for exact revision-fenced media identities. Still images and image sequences use real decoded source pixels, while WAVE audio uses exact decoded sample timing and ordered channel meaning.

## Architecture and integration

The existing Tauri project-media store remains the identity and freshness owner. One read-only `desktop_generate_media_preview` command validates project revision, media-library revision, media ID, and content fingerprint before releasing the store lock and entering bounded blocking work. Generated PNG data URLs are ephemeral consumer artifacts and never enter project persistence, derived-media attachments, source identity, timeline state, routing, playback, or export.

The application decodes supported still-image containers at the shell boundary, adapts their explicit RGBA8, sRGB, and straight-alpha meaning into `superi-image`, and delegates all aspect-fit scaling to its existing thumbnail contract. Image-sequence filmstrips choose at most six deterministic source indices in canonical order and apply the same lower scaling operation to every representative frame. The typed TypeScript bridge ignores late responses whose media ID or fingerprint no longer matches the selected item, replaces the selected browser thumbnail only when ready, and presents the full preview, filmstrip, and per-product unavailable state in the existing inspector.

WAVE audio opens through `superi-media-io::PcmContainerSource`, decodes through the in-tree `superi-codecs-rs::PcmBackend`, and passes every exact contiguous `AudioBlock` to `superi-media-io::preview`. The lower adapter rejects sample gaps, overlaps, format drift, and channel-layout drift before rasterization. The response retains start sample, sample rate, frame count, and ordered channel labels; no resampling, remixing, routing, muting, silence insertion, playback, or authored-state mutation occurs.

## Bounds and failure behavior

Source files, decoded image allocation, dimensions, PNG payloads, filmstrip frame count, decoded PCM bytes, channel count, and media operation time are bounded. Each thumbnail, filmstrip, waveform, and preview product is independently ready or unavailable with a user-safe reason. Unsupported video, compressed-audio, EXR, and DPX paths remain explicit unavailable products until an application-owned bounded decoder path can preserve their full media semantics.

## Verification

The focused native contract writes and imports one real PNG still, three real PNG sequence frames, a stereo 48 kHz 16-bit WAVE containing 262,145 frames, and an unsupported video source. It proves ready PNG data URLs, deterministic filmstrip order, exact sample range and channel order across the 1 MiB PCM packet boundary, stale revision and fingerprint rejection, explicit unavailable products, Tauri registration, typed bridge coverage, and the production React consumer. Existing import, media-library, offline-media, and derived-media contracts remain green, and strict TypeScript checking, the production Vite build, frontend tests, Rust formatting, map validation, and the repository checkpoint verifier are delivery gates.

## Remaining constraints

Preview artifacts are intentionally regenerated rather than persisted or attached as proxies. The current complete-block waveform adapter caps decoded PCM at 128 MiB, and the frontend generates only the selected item's bundle so large media libraries do not eagerly decode every asset. Video frame extraction and compressed-audio waveforms remain later decoder-session integration work rather than browser playback or guessed content.
