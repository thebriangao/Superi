# P1.W07.C004

## Outcome

Added a blocking frontend CI lane that installs an exact npm lockfile, runs strict TypeScript checking
independently, creates a production Vite bundle, and validates the generated hashed JavaScript entry.

## Architecture and integration

`.github/workflows/frontend.yml` owns the frontend gate without overlapping the cross-platform Rust
topology or Tauri validation assigned to neighboring checkpoints. The complete
`ci/frontend-smoke/` module is an explicit CI toolchain contract because the Phase 3 React
application is intentionally absent; it exercises a real typed browser entry and production Vite
consumer path without pretending that the product UI or editorial slice exists.

## Implementation process

The verified base was `a11cecdbf19ae1de90d94324abe844db49ed0c85`. A Node contract test was
written first and failed on the absent package, workflow, and bundle. The historical Vite 7.2.4
version was rejected after npm reported a direct high-severity advisory, then replaced with the
compatible fixed 7.3.6 release before the lockfile and implementation were finalized.

## Verification

Fresh frontend verification used `npm ci`, `npm run typecheck`, `npm run build`, and
`npm test`; all three contract tests passed and npm reported zero vulnerabilities. Negative controls
proved that an invalid TypeScript assignment fails with TS2322 and that a missing `dist/` artifact
fails bundle validation. Workflow YAML parsing, `cargo fmt --all --check`, locked workspace metadata,
and diff hygiene also passed. The complete locked Rust workspace unit, integration, and documentation
test suite passed with `CARGO_TARGET_DIR=/private/tmp/superi-shared-target` under the repository
resource gate.

## Delivery

This record ships in the same delivery commit as the workflow, frontend contract, lockfile, and
tests. The exact delivery SHA and `origin/main` containment are verified after commit creation and
recorded in the canonical checklist completion description.

## Remaining constraints

The real React and Tauri application, platform-native shell validation, hardware evidence,
malformed-input suites, and the first editorial slice remain owned by their later checkpoints. When
the real application enters the repository, this lane must run the same separate typecheck and
production-build gates against it rather than treating the CI contract as application coverage.
