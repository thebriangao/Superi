# P3.W04.C003 track management and control state

## Outcome

The editing workspace now creates, deletes, names, resizes, reorders, targets, locks, sync-locks, mutes, solos, and enables canonical timeline tracks through one durable command path. Every gesture is revision-fenced, history-backed, discoverable through labeled controls, and refreshed from the authoritative replacement snapshot instead of a second frontend edit model.

## Architecture and integration

`superi-timeline` owns one atomic `TrackMutation` batch for all eleven operations. Stable names and bottom-to-top order remain on `Track` and `Timeline`, while `TrackEditState` owns bounded lane height plus target, lock, sync-lock, mute, solo, and enable intent. Video, audio, caption, and data creation use explicit deterministic templates and caller-provided identities. Mute and solo are audio-only, a locked track rejects authored-item edits and deletion, and structural reconciliation preserves every surviving object, relationship, annotation, multicam reference, and control.

The strict timeline component advances to revision 2. Exact revision 1 and revision 0 wire models validate their original payloads and migrate missing controls to explicit defaults, while malformed current state remains rejected. Processing-owned enable, mute, and solo intent compiles as a typed track-output graph parameter. Height, targeting, and locks remain editorial intent and do not perturb processing state.

`superi-engine` publishes each track batch inside the existing compound project transaction. It recompiles retained graphs through the established three-way merge, records typed action evidence, and restores complete snapshots through undo and redo. Populated track deletion compares project-wide clip identity before and after the batch, then releases only clip-mix controls belonging to clips that disappeared. The outer transaction rolls editorial, audio, graph, history, and durable state back together on failure.

`superi-api` exposes strict tagged track DTOs through the unchanged `superi.project.command.execute` method and reports `tracks_mutated` evidence. The generated TypeScript artifact derives from that Rust authority. The native dispatcher routes editor-state reads and project commands through the active durable `LocalProjectHost`, then publishes the canonical `superi.project.state.changed` event in its ordered stream.

`ApplicationProvider` owns unique transaction identity, the visible project revision fence, command execution, and replacement-state refresh. `TimelineWorkspace` receives only a mutation callback. Its add, inline name, height, order, target, lock, sync-lock, audio mute and solo, enable, and guarded delete controls expose pending and failure state with titles and ARIA labels while lane geometry comes directly from canonical track height.

## Failure and compatibility behavior

Unknown or duplicate identities, invalid positions or heights, illegal nonaudio mute or solo intent, locked content edits, locked deletion, stale revisions, malformed public requests, absent active projects, and durable host failures reject without partial publication. Existing revision 0 and revision 1 timeline documents load with the former 72 pixel visual default and neutral control state. Existing public method and event names, media semantics, source and record timing, item identity, grouping, linking, synchronization, source monitor behavior, and direct graph-edit reconciliation remain intact.

## Verification

The complete timeline suite proves all eleven controls, atomic rollback, locked edit enforcement, strict migrations, deterministic round trips, and distinct compiled output intent. Engine contracts prove one project publication, graph reconciliation, exact evidence, populated-track clip-mix cleanup, undo, redo, and late-failure rollback. Public API contracts prove strict tags, catalog parity, durable execution, generated bindings, and command evidence. The full native suite proves the real durable desktop route and canonical event, while strict TypeScript checking, 25 frontend contracts, and the production Vite build prove the application owner and transport-free consumer.

## Remaining constraints

Track height is authored presentation intent rather than processing state. Target, lock, and sync-lock guide editing policy but do not independently schedule playback. Track mute, solo, and enable are now retained and compiled as output intent; runtime graph evaluation and final audio or video mixing remain owned by later execution checkpoints. Drag reordering and continuous drag resizing can layer on the same atomic mutation surface without creating a parallel command owner.
