# P3.W04.C007 exact timeline editing gestures

## Outcome

The production timeline canvas now exposes ripple, roll, slip, slide, razor, trim, and extend as precise editorial tools, together with synchronized ripple delete, gap insertion, and gap closure. Every direct gesture shows its exact target and affected material before one atomic publication, and selection-only clicks cannot become accidental edits.

## Architecture and integration

`superi-timeline` remains the owner of editorial mutation semantics, identity propagation, grouping, synchronization, transition cleanup, and inherited user intent. The desktop `timeline-editing` compiler consumes the strict canonical timeline projection and produces immutable public `TimelineEditOperation` batches. It does not mutate projected state, approximate a lower operation, or publish a second authored timeline model.

All record and source calculations use canonical integer values and explicit rational clocks. Display coordinates first resolve to one exact edit-frame point, then rescale exactly into each affected track or source clock. Inexact cross-clock edits fail before identity allocation or publication, preventing a rounded display coordinate from changing authored timing.

The shared desktop project executor publishes every plan as one project transaction. Canonical state refresh follows successful persistence, while a rejected or stale command leaves the visible model unchanged. Existing engine-owned history supplies immediate undo and redo instead of a component-local reversal stack.

## Editorial gestures and visible state

The workspace exposes all seven tools with explicit start or end targeting and explicit ripple or roll extend mode. Pointer drafting starts only after a four-pixel threshold, continuously recompiles an exact snap-aware preview, and recompiles once more from the release coordinate before publication. Razor is a direct exact-point action; every other direct tool supports drag preview, frame nudges, Escape cancellation, pending exclusion, and one release publication.

Roll and slide require exact adjacency and retain the total surrounding record span. Slip changes only the clip source range. Razor assigns one typed identity to the new right fragment. Trim creates a typed gap only when an inward edit needs one and consumes only a real adjacent gap on an outward edit. Ripple and ripple extend carry deterministic adjustments across every synchronization-locked companion track.

Ripple delete and gap closure compile ordered per-track extracts across the primary and synchronization-locked tracks. Gap insertion compiles ordered per-track inserts with track-clock durations and typed gap identities. Fragment identities are allocated only for the exact boundary case consumed by the lower operation. The consequence preview names the plan, exact time, operation count, and affected objects before execution.

## Failure handling

Missing tracks or objects, locked primary or synchronized tracks, transition targets, empty ranges, invalid adjacency, zero-duration results, unavailable adjacent gaps, malformed exact integers, mismatched clocks, inexact rescaling, unsafe API coordinates, and invalid generated identities fail before publication or identity allocation. A locked track remains selectable for inspection while its timing actions stay disabled. Pending execution blocks a second gesture, and a failed command reports the error while preserving canonical state and the previous authored history.

## Verification

The focused compiler contract covers all seven tools, both extend modes, ripple synchronization, mixed 24 fps and 48 kHz tracks, fractional 24000/1001 display timing, typed fragments, ripple deletion, gap insertion, gap closure, immutable plans, locked-track rejection, and failure before identity allocation. Workspace contracts cover the visible catalog, exact preview, drag threshold, release-time recompilation, cancellation, executor injection, and selection integration. The complete frontend suite, native retained-session and transport contracts, lower timeline operation contracts, engine transaction and API editor contracts, updated codebase maps, and repository checkpoint verifier exercise the real consumer path.

## Remaining constraints

The compiler intentionally emits only operations already owned by the public editor contract. Transitions remain edited through adjacent timed objects, authored selection remains in the application selection owner, snapping remains a transient exact query, and project history remains engine-owned. Any future gesture must reuse those same owners instead of adding frontend mutation, persistence, identity, synchronization, or reversal authority.
