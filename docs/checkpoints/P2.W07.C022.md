# P2.W07.C022: supported local scripting runtime

Superi supports a dependency-free local JSON scripting language named `superi-json`. Version
`1.0.0` is interpreted by `superi.project.script.run` through the same `ProjectEditorApi` that owns
interactive public project commands. The runtime does not introduce a second project owner, a
dynamic method router, a network dependency, or access to engine internals.

## Source files and identity

The conventional source suffix is `.superi-script.json`. Loading a file is caller-owned: the public
request accepts the exact UTF-8 source text and its required lowercase SHA-256 digest. The runtime
does not open paths, search directories, resolve imports, or alter source whitespace before hashing.

One complete read-only script is:

```json
{
  "language": "superi-json",
  "language_version": "1.0.0",
  "script_id": "daily.conform-01",
  "expected_initial_project_revision": 0,
  "steps": [
    {
      "method": "superi.editor.state.get",
      "params": {
        "transaction_id": "daily-conform-state"
      }
    }
  ]
}
```

Calculate the digest over the file's exact bytes:

```bash
shasum -a 256 daily.conform-01.superi-script.json
```

Then provide the exact file text as `source` and the 64-character digest as
`expected_source_sha256` in `RunProjectScript`. The generated TypeScript artifact exposes
`RunProjectScript`, `ProjectScriptProgram`, `RunProjectScriptResult`, and the method map entry needed
to make the same typed request from a desktop consumer.

`script_id` is a caller-owned trace identity. It begins with a lowercase ASCII letter, contains at
most 128 bytes, and otherwise accepts lowercase ASCII letters, digits, `.`, `_`, `-`, and `:`.

## Supported steps

Version 1 has a closed step union:

- `superi.project.command.execute` accepts the existing `ExecuteProjectCommand` request and returns
  the existing `ExecuteProjectCommandResult`.
- `superi.editor.state.get` accepts the existing `GetEditorState` request and returns the existing
  `GetEditorStateResult`.

Unknown methods, fields, language names, and language versions are rejected. Scripts cannot invoke
host filesystem APIs, plugins directly, arbitrary JSON-RPC methods, network clients, subprocesses,
or code evaluation. The generic project command remains the one stable mutation surface, including
timeline, graph, media, mix, extension, undo, redo, and inspection behavior that it already owns.

## Bounds and validation

The runtime validates the complete request before it executes a source step:

- exact source size is at most 1,048,576 UTF-8 bytes;
- one program contains from 1 through 256 ordered steps;
- one script identity contains at most 128 UTF-8 bytes and must satisfy the canonical grammar;
- JSON nesting depth is at most 128 values;
- duplicate object names are rejected at every nesting level;
- every step is decoded through its strict existing request type;
- every transaction identity and compound project action is converted and validated up front;
- the expected source digest must match the exact source bytes;
- `expected_initial_project_revision` must match the authoritative project before any source step.

These limits are part of scripting schema `1.0.0`. A caller must not assume that a future incompatible
language version will be accepted by this runtime.

## Deterministic execution and conflicts

The runtime captures authoritative complete editor state, checks the initial revision fence, and
executes valid steps in source order through `ProjectEditorApi::execute`. Successful responses are
retained as an ordered typed trace with zero-based source indexes. The result also records script and
source identity, initial and final project revisions, initial and final semantic hashes, final
status, the first failed index when present, a sanitized public failure, and whether authored effects
were committed.

The same exact source and equivalent complete initial runtime state produce the same interpretation
and serialized trace. Source whitespace is meaningful because it changes the exact digest. Project
schema meaning, media IDs, referenced path identity, fingerprints, semantic hashes, and revisions
continue to use their existing canonical project contracts rather than a script-specific encoding.

An initial revision conflict returns `rejected`, completes no source steps, publishes no authored
event, and commits no effects. A failure reached during execution returns `stopped`, records the
failed step index and last valid project resource, preserves every successfully committed prefix
step, and does not execute the suffix. The public error contains only reviewed, user-safe data.

A whole script is intentionally not a transaction. When several authored actions must succeed or
fail together, place them in one `superi.project.command.execute` apply step, whose existing compound
transaction contract is atomic. Undo and recovery continue to use the existing project history and
recovery systems.

## Permissions and events

The runtime declares payload-dependent filesystem and plugin permission kinds because nested generic
project actions can require them. Before dispatch, it parses every step and aggregates the exact
nested permission requirements for host authorization. Each nested command is authorized again by
the normal public command path when it executes. The scripting runtime itself requests no source-file
access and grants no capability.

Successful nested project commands preserve their original transaction identities, command
sequences, and `superi.project.state.changed` events. The runtime does not drain, replace, or wrap
those events in a script-specific mutation event. Consumers continue to poll or drain the existing
bounded event surface.

## Durability and recovery

After execution, the authoritative `ProjectDocument` remains the durable source of truth. Publishing
that snapshot through `ProjectDatabase` preserves the existing schema version, project and media
identity, referenced paths, fingerprint evidence, timeline and graph meaning, extension records,
revision, and semantic integrity hash. Reload, integrity inspection, recovery comparison, and
recovery restore therefore behave exactly as they do for interactive commands.

The runtime never claims rollback for a committed prefix. A caller can inspect the final trace,
persist the last valid snapshot, use normal undo for retained authored history, or use normal crash
recovery when a durable recovery candidate exists.

## Public versions

`superi.project.script.run` has scripting schema `1.0.0`. Its addition advances the complete public
catalog and permission catalog to `1.2.0`; existing domain methods retain their own versions. The
catalog-driven Rust and TypeScript surfaces are canonical, so consumers should discover the method
and version from the public schema instead of maintaining a parallel method list.
