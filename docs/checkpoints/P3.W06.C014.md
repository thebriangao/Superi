# P3.W06.C014

## Outcome

Implemented one persistent background-job center for proxy, analysis, cache, save, and export
activity in the production desktop application. Proxy, analysis, cache, and save work now enters a
bounded shell-local journal around the existing real operation adapters, while export progress
continues to project the authoritative public engine queue. The center retains truthful operation
status, failure guidance, same-session retry when the original closure still exists, terminal
dismissal, category and status filtering, and restart evidence without moving editable project or
cache payloads into presentation storage.

## Architecture and integration

`app/src/background-jobs.ts` owns a schema-1 immutable metadata journal with monotonically allocated
identities, a 64-receipt bound, explicit running, completed, failed, and interrupted states, and
strict restoration validation. Capacity pressure removes only terminal history and never hides
unfinished work. Work results and thrown failures remain unchanged for callers, retry creates a new
receipt through the same adapter, persisted running work becomes interrupted after restart, and no
closure or operation payload crosses the persistence boundary.

`app/src/project-lifecycle.ts` wraps only existing asynchronous project operations. Project save,
save-as, and recovery restore enter the save category; generated previews enter cache; content
analysis enters analysis; and derived proxy or optimized-media creation enters proxy. The wrappers
delegate to the same Tauri commands and return the same replacement snapshots, so the project,
media-library, cache, and engine owners remain unchanged.

`app/src/desktop-shell.ts` synchronizes the journal with the primary shell presentation and resumes
the native client sequence after hydration. `app/src-tauri/src/desktop_shell.rs` advances the private
presentation record to schema 4, migrates schema 1 through 3 workspace and shortcut state, validates
every receipt, and recovers a corrupt job section independently from valid workspace and shortcut
state. Storage contains bounded operational metadata only, never project documents, cache products,
media bytes, job closures, or engine results.

`application-presentation.ts`, `application-presentation.tsx`, and `App.tsx` combine local receipts
with the existing public export snapshots in one visible center. The center supports category and
status filters, exposes retry only for same-session local failures, permits dismissal only for local
terminal receipts, includes active local work in safe-close busy state, and leaves export scheduling
and control with the public engine queue.

## Implementation process

Focused frontend tests first failed because the background journal and presentation adapter did not
exist. Focused native tests then failed because the shell contract had no background-job DTO or
persistence field. Implementation added the bounded journal, real lifecycle wrapping, schema-4
native validation and migration, unified presentation, accessible filters and actions, and semantic
theme-token styling without adding a timer, polling loop, second scheduler, or new public API.

## Verification

Focused frontend contracts cover running and terminal publication, unchanged work results, classified
failures, same-session retry, restart interruption, strict restore validation, terminal dismissal,
bounded history, full-capacity unfinished-work protection, lifecycle category routing, native bridge
hydration, and application presentation. Focused native contracts cover schema-4 persistence,
schema-1 through schema-3 migration, strict timestamp and receipt validation, independent corrupt-job
recovery, and unchanged live state after rejected synchronization. Strict TypeScript, the production
Vite build, the complete frontend suite, the Tauri desktop-shell contract, codebase-map validation,
and the deterministic checkpoint verifier provide the widening local integration floor.

## Remaining constraints

Only same-session local failures can retain a retry closure. Restored failures and interrupted work
remain actionable evidence but must be restarted from their owning workspace. Export rows are a
read-only projection of the public engine queue because the current desktop transport does not expose
its control surface here. The journal deliberately persists no operation inputs, project data, media,
cache artifacts, generated results, or arbitrary error detail.
