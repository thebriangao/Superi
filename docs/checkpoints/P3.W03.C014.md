# P3.W03.C014 source monitor loading and marks

## Outcome

The editing workspace now loads project media into one retained source-monitor session, exposes exact engine and source state, seeks on the source clock, and stores reversible in and out marks against the source fingerprint on which they were authored. Container audio and video use the real engine-owned source registry, while stills and image sequences use their verified project paths and exact imported ranges.

## Architecture and integration

`superi-engine::media::source_backend_registry` reuses the engine's single source-backend assembly without initializing unrelated decoder or encoder runtimes. It exposes MKV/WebM, MP4/MOV, MXF, and PCM source loading independently from codec availability. The complete media registry continues to compose the same source registrations with its codec backends for decode and encode consumers.

`DesktopProjectState` owns one retained source monitor behind a mutex. Load validates project revision, library revision, media identity, stored content fingerprint, accepted source-monitoring state, and current source bytes before doing blocking probe and open work, then rechecks every fence before publication. Seek requires the current monitor revision and exact reduced rational time, uses `SeekMode::Exact` for opened container sources, and validates inclusive imported ranges for stills and image sequences. A later exact source scan that proves changed bytes makes the retained session stale and fences further seek or mark actions until the source is reviewed. Project close, project replacement, and save-as release the retained source.

Each media-browser item can persist optional `SourceMonitorMarks`. Mark intent contains exact rational in and out coordinates plus the source fingerprint, remains backward-compatible with older sidecars, survives reimport of the same stable media identity, and becomes visibly stale after source identity changes. Set and clear operations clone and validate the complete library, reject in after out, publish the sidecar atomically, and advance the media-library and monitor revisions only after success.

The TypeScript bridge exposes replacement snapshots and optimistic load, seek, mark, and unload requests. `SourceMonitor` composes the existing role-colored native source viewport without moving state into the workspace projection. Its controls show empty, ready, or stale state, backend, container, selected stream, codec, exact coordinate and duration, mark freshness, and actionable failures. The interface states explicitly that a retained source session is separate from decode and native GPU viewer presentation.

## Failure and compatibility behavior

Unsupported containers, offline paths, changed fingerprints, scanner-confirmed changed bytes, stale project or library revisions, stale monitor revisions, invalid or noncanonical timebases, out-of-range seeks, removed media, and reversed marks reject without partial durable publication. Failed load leaves any prior retained session intact. Failed seek leaves the current coordinate and monitor revision unchanged. Failed mark publication leaves the media library, sidecar, and monitor revision unchanged.

Existing media import, relink, proxy, search, metadata, annotation, preview, offline, and native viewport contracts retain their request shapes and ownership. Source monitor loading does not claim packet decode, frame presentation, playback transport, cache population, proxy generation, or GPU rendering.

## Verification

The focused source-monitor contract opens a real mono 48 kHz WAVE through the engine source registry, proves exact seeks, publishes and restores fingerprint-bound marks, rejects a reversed mark transaction without advancing the library, proves an exact changed-byte scan makes the session stale and fences another seek, and unloads the retained source. It also imports a real three-frame PNG sequence, proves its 24 fps inclusive range, accepts the last frame, rejects overrun without advancing monitor state, and locks Tauri registration, typed bridge coverage, workspace purity, and honest presentation wording. A focused engine test proves all four source backends construct without codec runtime initialization. Existing media import, identity, preview, removable-media, and viewer contracts, strict TypeScript checking, frontend contracts, the production Vite build, Rust formatting and linting, map validation, and the repository checkpoint verifier are delivery gates.

## Remaining constraints

The monitor retains a source for inspection and exact positioning but does not decode packets into frames or audio blocks. The existing native GPU viewport continues to present its canonical engine-provided texture independently. Raw FLAC, MP3, AAC, EXR, and DPX monitor sessions remain unavailable until their owning source or still-image adapters can preserve the required source semantics without guessed browser behavior.
