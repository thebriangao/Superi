# Superi Research and Planning

## Entry gate

Begin only after repository synchronization, a verified Google Docs claim, and initial map
validation. Create:

- `plans/<checkpoint-id>/planning.md`
- `plans/<checkpoint-id>/execution.md`

Create no other checkpoint plan files.

## Read the real scope

1. Record the synchronized base revision and exact claim.
2. Read the complete live main specification and surrounding phase, workstream, subsystem, and prior
   completion context.
3. Read root `AGENTS.md` and every required repository-owned skill in full.
4. Read the global map and affected maps in full.
5. Read every file that may change plus relevant interfaces, callers, consumers, manifests, schemas,
   tests, fixtures, migration sources, and governing documents through EOF.
6. Record the complete reading inventory in `planning.md`.

Do not infer scope from a convenient file, a stale checkpoint note, or an audit summary.

## Research decision

Use repository and checklist evidence first. Research externally only when a material product,
platform, standards, licensing, or architecture question remains.

When research is needed:

- use current public primary sources
- compare at least two sources for consequential product claims
- distinguish source fact from Superi inference
- record URLs, access date, evidence, and decision
- never launch, control, or capture a third-party creative desktop application on the user's
  computer

## Product and visual planning

For a Phase Infinity surface:

1. Identify the user task, canonical state owner, automation route, selection domain, focus behavior,
   failure states, accessibility semantics, and consumer proof.
2. Read `../superi-interface-design/SKILL.md`, then inspect relevant current interface captures and
   adjacent surfaces against its locked decisions.
3. Brainstorm at least three original directions when interaction or visual judgment is material.
4. Compare directions against task speed, hierarchy, density, accessibility, consistency, rendering
   cost, and canonical ownership.
5. Choose one direction and record rejected alternatives plus reasons.
6. List every needed icon and decide reuse versus new design.
7. Define private capture states and crops before implementation.

Published tools may inform task patterns, but never copy their artwork, layout, labels, measurements,
or skin.

## Architecture plan

`planning.md` must define:

- assignment and non-goals
- base revision and claim
- reading inventory
- current behavior and gap
- canonical owners and API routes
- dependency direction
- selected product and visual direction
- state, input, focus, semantics, and accessibility
- ordered implementation slices
- error, degraded, migration, and compatibility paths
- icon and asset work
- map updates
- focused, subsystem, real-consumer, capture, and final proof
- risks and rollback-safe recovery

`execution.md` must begin with:

- base revision
- live command queue
- expected failing proof
- implementation checklist
- verification checklist
- artifact and screenshot destinations

## Exit gate

Re-read both plan files in full. Planning is complete only when another implementation choice would
not be required to begin the first source slice, every affected owner is identified, and every
claimed product outcome has a real proof.
